# 🎧 Flutterアプリに「音声認識モデルを入れる」だけの話

Flutterで音声メモアプリのようなものを作っていて、[「録音」](https://qiita.com/nobumory/items/6bc964a69c94a007346e)と[「しゃべった部分の区切り」](https://qiita.com/nobumory/items/764ed02b0cf2ccd7d028)まではできました。

次はいよいよ本命、「音声を文字にする（音声認識）」です。

音声認識には「**モデル**」というファイルが要ります。AIの“脳みそ”が詰まった、あの大きいやつです。

でも「モデルファイルを手に入れて、アプリに入れるだけでしょ」と高をくくっていたのですが……この『アプリに入れるだけ』で、週末の午前中を溶かしました。

この記事では、**モデルを初めてアプリに組み込む人**に向けて、「モデルをアプリに入れるって具体的に何をしていて、なんでそんなに大変なの？」を噛み砕いて書きます。細かいコードの話というより、「そもそも何が起きているのか」をざっくり掴むのが目的です。

（音声認識をアプリに入れる**全体像**は[別記事「ゼロ知識からのFlutterで音声認識実装」](https://qiita.com/nobumory/items/1e7e25f3395659d2eff6)にまとめています。本記事は、その中で「一番の山」と書いた**モデル配布**を深掘りする回です。）



## 30秒でいうと

忙しい人向けに、先に結論だけ。

- 音声認識には「**モデル**」という賢いファイルが要る。これは **AIの“脳みそ”が詰まった、1個の大きなファイル**。
- このモデルが **数百MB級（今回のは約625MB）とデカい** ので、スマホアプリに入れるのが地味に大変。
- しかも **iOSとAndroidで「置き方」が全然違う**。だから「**モデルのパスを返してくれる薄い抽象化レイヤ**」を1枚挟んでおくと、後がすごくラクになる。



## そもそも「モデル」って何？

ひとことで言うと、モデルは **AIの“脳みそ”が詰まった1個の大きなファイル** です。

たとえば音声認識モデルは、「**音を入れると、文字を返してくれる脳**」だと思ってください。中身は人間には読めない数字の塊ですが、使う側からすると「音 → 文字」の変換をしてくれる箱です。

```text
[ 音声データ ] → 🧠（モデル） → 「今日はいい天気」
```

この脳は、あらかじめ大量の音声で“訓練”されています。その訓練の成果が1個のファイル（今回のものだと約625MB）に固められていて、これをアプリに持たせると、ネットに繋がなくてもスマホの中だけで音声認識ができます。

> ちなみに、この脳を**実際に動かす側**が「音声認識エンジン」です。エンジンは **アプリに組み込むライブラリ（＝コード側）**、モデルは **それが読み込むデータ（＝ファイル）**、という棲み分け。**CDプレイヤーとCD**の関係に近いです（プレイヤー＝エンジン、CD＝モデル）。僕はエンジンに `sherpa-onnx` を使っています（このあたりは[別記事「Flutterでsherpa-onnxを触ってみた」](https://qiita.com/nobumory/items/52b2f145bd204c10b6d1)に書きました）。

## 今回アプリに入れる「ファイル」は3つ

「音声認識モデル」と一口に言っても、入れるファイルは1つではありません。今回、端末に用意したのは次の **3つ** です。

| ファイル | 役割 | 大きさ |
|---|---|---|
| 認識モデル本体 | 音声を受け取って“文字のもと”を出す脳 | 約625MB |
| 変換表（tokens） | 本体の出力を文字に直す対応表 | 約28KB |
| 区切り用モデル（VAD） | 「話し始め・話し終わり」を見つける | 約0.6MB |

- **本体（約625MB）** … この記事の主役。「音 → 文字」を担当する脳。デカいのはこいつです（…が、実は直接“文字”を出していません。次の節で）。
- **変換表（tokens）** … 本体とペアで使う小さなテキスト。これも次の節で説明します。
- **区切り用モデル（VAD）** … マイクの音から「しゃべっている部分」だけを切り出す、別の小さなモデル。これも“モデル”の一種です（区切りの話は[別記事「録音した音を発話ごとに区切ってみた #VAD」](https://qiita.com/nobumory/items/764ed02b0cf2ccd7d028)に書きました）。

置き場所で本当に悩むのは、デカい **本体（625MB）** だけ。残り2つ（変換表・VAD）は小さいので、**普通にアプリに同梱すればOK**（Gitにも入れられるし、後で出てくる“二重持ち”問題も気になりません）。なので以降は、ほぼ **本体625MBをどう置くか** の話だと思ってください。



## そのモデル、どこで手に入れるの？

ここまで「モデルを手に入れたら」とサラッと書いてきましたが、そもそもこの625MBの脳、どこから持ってくるのでしょう。

まず大前提として、**でかすぎてGitには入れられません**。コードと一緒にリポジトリで管理…とはいかないので（数百MBのファイルを毎回扱うのは現実的じゃない）、**配布元からダウンロードしてくる**形になります。今回は、使っている音声認識ライブラリの作者本人が公開しているモデル置き場（**Hugging Face** という、AIモデルの配布でよく使われる場所）から落としました。

ここでのコツは、**「特定のバージョンを名指しで」落とすこと**です。Hugging Face のモデル置き場は中身が **Git で管理されている**ので、**Gitのコミットを指定するのと同じ感覚**で「このバージョン」と固定できます。

「最新（`main`）」を指す“動く矢印”のまま落とすと、配布元がファイルを差し替えたときに、手元に来る中身が勝手に変わってしまいます。そうではなく、特定のコミットに固定しておくと、

- **誰がいつ落としても、まったく同じファイル**が手に入る
- ダウンロードがちゃんと終わったかの**サイズ確認（後述）もブレない**

という安心が得られます。そしてこの「落とす → 確認する → 置く」を毎回手でやると間違えるので、**小さなスクリプト1本にまとめて**、コマンド一発で済むようにしました。



## ちょっと意外：モデルは「文字」じゃなく「番号」を返す

ここは最初びっくりした話です。しかもこれ、音声認識に限った話ではなく、**文章を扱うAIモデル全般（LLMや翻訳モデルなど）に共通する性質**だったりします。

「音 → 文字」の脳と言いましたが、**実はモデルが直接出してくるのは文字ではありません**。出てくるのは「**番号の列**」です。

```text
モデルの出力:  [ 5, 102, 8, 240, ... ]   ← これだけ見ても意味不明
```

この番号を文字に直すための「**番号 → 文字の変換表**」が、モデルとは別に必要になります。これは小さなテキストファイル（数十KB）です。

```text
変換表:  5 → こ  /  102 → んにちは  /  8 → 。 …
```

つまり音声認識には、

- **大きいファイル**（脳本体・音 → 番号） … 約625MB
- **小さいファイル**（変換表・番号 → 文字） … 約28KB

の **2つがワンセット** で必要、ということです。そして、この2つは**必ず同じバージョンで揃えないといけません**。番号の振り方はモデルごとに違うので、別バージョンの変換表を混ぜると、見事に文字化けします。



## 「アプリに入れる」だけじゃ、足りない

ここがこの記事の本題です。

モデルファイルを手に入れたら、あとはアプリに含めるだけ……と思いますよね。僕もそう思っていました。でも、エンジンには**ちょっと不便な性質**があります。

> **エンジンは、モデルの「実ファイルのパス」（`/data/.../model.onnx` のような、端末上の本物のファイルの場所）を渡さないと読み込めない。**

ここで困ったことが起きます。Flutterには画像などをアプリに同梱する「**アセット**」という仕組みがありますが、**アセットに入れたファイルには、この実ファイルのパスがありません**。

アセットはアプリ本体（`apk`/`ipa`）の中に **梱包されたまま**入っていて、OSが扱う1個のファイルとして **展開されていない** からです。`rootBundle` でバイト列としては読めても、「端末のこのフォルダにある」と言えるパスは無い。だからそのままではエンジンに渡せません。

なので「アプリに含める」だけでは足りず、**エンジンに渡せる実ファイルのパスがある場所にモデルを用意する**、というのがこのステップのゴールです。「置く」とは、ただ含めることではなく、**パスを用意すること**なんです。



## デカすぎ問題（数百MB）

そして、モデルが **約625MBとデカい** ことが、話をさらにややこしくします。

普通のやり方（Flutterのアセットに入れる）には落とし穴があります。前の節のとおりアセットにはパスが無いので、使う前に**実ファイルとして別の場所へまるごとコピー**して、はじめてパスが手に入ります。

すると、**1台のスマホの中に**、

- ① アプリに埋め込まれた 625MB
- ② パスを得るためにコピーした 625MB

の **同じファイルが2つ** できてしまいます。その端末だけで合わせて約1.25GB（iOSとAndroidの合算ではなく、**1台でこれ**）。さすがに無駄ですよね。

しかも、この事情は **iOSとAndroidで違います**。

- **iOS** … 「コピーせず実ファイルのパスが取れる」入れ方がある（いわゆる **バンドルリソース**／iOSの *Bundle resource*）。`Bundle.main` から直接パスが取れるので、二重持ちを避けられる。
- **Android** … アプリ本体のサイズに **約200MBの制限**（Google Play の上限）があり、625MBは**そのままでは入らない**。そこで **Play Asset Delivery（PAD）** ―― インストール直後に裏で自動ダウンロードする仕組み ―― を使う必要があります。ただ重いので、**今回は「開発中は手で端末に置く」だけ**にして、本番の自動配信は別の回に回しました（手置きでハマった実話は最後のコラムで）。

要するに、**同じモデルなのに、iOSとAndroidで「どう届けて、どこに置くか」が全然違う**わけです。



## コツ：「置き方の違い」は隠すと幸せ（宅配ボックスの発想）

ここまでで、「届け方がOSごとにバラバラ」という事実が出てきました。

でも、モデルを**使う側**からすると、そんな事情はどうでもよくて、知りたいのはただ一つ、「**結局、パスはどこ？**」だけです。

そこで、**配置の違いを内側に隠して「パスだけ」を返す薄い抽象化レイヤ**を1枚かませます。宅配ボックスのイメージです。

```text
配達する側（iOSの同梱 / Androidのダウンロード / 開発中の手置き…）
        ↓ いろんな届け方
   📦 抽象化レイヤ（パスはいつもここで受け取る）
        ↓ 使う側は
   「このパスを読むだけ」
```

- 用意する側：どんな届け方でも、決まったやり方で「パス」を返す。
- 使う側：そのパスを受け取って読むだけ。**どうやって届いたかは気にしない。**

こうしておくと、たとえば後でAndroidの届け方（手置き → 本番の自動配信）を変えても、**使う側のコードは1行も変わりません**。差分は全部このレイヤの内側に閉じ込められます。

僕はこのレイヤ1枚に「本体・変換表・VAD」の3つのパス解決をまとめて任せました。使う側は、これに聞けば3つのパスが返ってくる、という形です。



## ちゃんと置けたか、どう確認する？

大きいファイルを扱うと、**コピーやダウンロードが途中で止まって、中途半端なファイルが残る**ことがあります。これに気づかず使うと、エンジンの奥でいきなりクラッシュして、原因が分からず詰みます。

そこで、置いたあとに簡単なチェックを入れておきます。といっても難しいことはせず、

> **「ファイルがあるか＋サイズがちょうど合っているか」だけ見る。**

これだけです。途中で切れたファイルはサイズがズレるので、**サイズ照合だけでほぼ全部はじけます**。もっと厳密な方法（ハッシュ照合）もありますが、個人開発ではそこまでの手間に見合わないので、シンプルさを優先しました。

## まとめ

「モデルをアプリに入れる」は、最初は単純なコピー作業だと思っていました。でも実際は、

- モデルは **AIの脳みそが詰まった大きなファイル**（しかも変換表とワンセット）
- エンジンは **実ファイルのパスでしかモデルを読めない**
- モデルが **デカくて、OSごとに置き方が全然違う**
- だから **「パスを返す抽象化レイヤ」を1枚挟んで、違いを隠す**

という、地味だけどちゃんと設計が要る作業でした。

ここを最初に整えておくと、後で「届け方」を変えても使う側は無傷で済みます。**“置くだけ”に見えて、実は土台づくり**だったんだな、というのが今回の学びでした。
