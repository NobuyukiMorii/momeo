Flutterで音声メモアプリのようなものを作ろうとしていて、まずは「録音」の部分を調べていました。

音声を文字にする前に、そもそもマイクから音を取り込まないと何も始まりません。

その「マイクから音を取り込む」役割を担うのが `record` というパッケージです。

最初は「録音するだけのパッケージでしょ？」くらいに思っていたのですが、調べていくうちに、そもそも「録音する」とはどういうことなのか、音がデータになるとはどういうことなのか、というあたりが自分の中で曖昧だったことに気づきました。

この記事では、マイク・録音・PCMといった言葉が全然分からない状態からでも読めるように、`record` で何をしているのかを整理してみます。

細かいコードの解説というより、「そもそも何をしようとしているのか」を分かるようにするのが目的です。

## 30秒でいうと

`record` は、**マイクの音を「データ」として取り出すための録音パッケージ**です。

ポイントは、`record` は**音を取るだけ**で、音を文字にする（音声認識する）機能は持っていない、という点です。

音声メモアプリ全体で見ると、処理はだいたいこういう流れになります。

```text
マイク → 録音 → 区切り → 文字にする → メモとして保存
        ↑
        record はここだけ
```

`record` が担当するのは、一番最初の「マイクの音をデータにする」ところだけです。

その先の「話している部分を切り出す」「文字にする」は、別の道具（VADや音声認識モデル）の仕事になります。

なので `record` は、**音声処理の入り口**だと思ってもらうのが正確です。

## そもそも何をしようとしているのか

音声メモアプリでやりたいことは、最終的にはシンプルです。

**しゃべった内容を、文字にしてメモとして残す。**

ただ、これを実現するには、いくつかの段階を踏む必要があります。

| 段階 | やること | 担当 |
|---|---|---|
| 1. 録音 | マイクの音をデータにする | `record` |
| 2. 区切り | 話している部分だけを切り出す | VAD |
| 3. 文字化 | 音声を文字にする | 音声認識モデル |
| 4. 保存 | 文字をメモとして残す | アプリ本体 |

この記事で扱うのは、**1番目の「録音」だけ**です。

ここがちゃんと動かないと、2番目以降は全部成り立ちません。

「マイクの音が、ちゃんとデータとして取れている」——まずはこれを確実にするのが `record` の役割です。

## 前提：そもそも「音をデータにする」とは？

`record` を理解するには、「音がどうやってデータになるのか」を先に押さえておくと、一気に分かりやすくなります。

ここが曖昧なまま録音のコードを読むと、出てくる数字の意味が全然頭に入ってきません（僕がそうでした）。

### 音はもともと「波」

マイクが拾う音は、空気の振動です。

これは、なめらかに上下する「波」として表せます。

```text
音の波:  ╱╲    ╱╲
        ╱  ╲  ╱  ╲
```

でも、コンピュータはこの「なめらかな波」をそのままでは扱えません。

なめらかな波は無限に細かいので、どこかで「数字の列」に変換する必要があります。

### デジタル化＝「一定間隔で波の高さを測る」

そこでやるのが、**一定の間隔で波の高さをパッ、パッ、と測って数字にする**ことです。

```text
音の波:  ╱╲    ╱╲
        ・・・・・・・・   ← この「・」1個で波の高さを1回測る
```

この「1回測った値」を **サンプル** と呼びます。

そして、こうして測った数値をただ順番に並べたものが **PCM** と呼ばれる、一番素の音声データです。

PCMは、圧縮も加工もしていない「生の波形」だと思ってください。

### 録音で決める3つの設定

音をデータにするとき、決めることが3つあります。

`record` でも、この3つを設定します。

| 設定 | 何を決めるか | たとえると |
|---|---|---|
| サンプリングレート | 1秒間に何回 波の高さを測るか | パラパラ漫画の**コマ数** |
| ビット深度 | 1回の値を どれだけ細かい数字で記録するか | 写真1枚の**解像度** |
| チャンネル数 | 音のデータを 何本持つか | **モノラル / ステレオ** |

音声メモアプリでは、だいたい次の組み合わせを使います。

- **サンプリングレート：16kHz**（1秒間に16000回 測る）
- **ビット深度：16bit（PCM16）**（1回の値を -32768〜32767 の整数で記録 = 2バイト）
- **チャンネル：モノラル**（音のデータは1本）

この `16kHz / PCM16 / モノラル` という設定は、後で出てくるので頭の片隅に置いておいてください。

### ちょっとだけ補足：紛らわしい「16」

`16kHz` と `PCM16`、どちらにも「16」が出てきますが、意味は別物です。

- **16kHz** … 1秒に **16000回** 測る（時間の細かさ）
- **PCM16** … 1回の値を **16ビット** で記録する（音の大きさの細かさ）

数字が同じなのは偶然で、片方は「測る回数」、もう片方は「1個あたりの細かさ」です。

ここは僕も最初ごちゃっとしたので、念のため。

### モノラルとステレオ

チャンネル数は「音のデータの流れが何本あるか」です。

- **モノラル（1本）** … 電話の声のように、左右の区別がない音
- **ステレオ（2本）** … 音楽のように、左から楽器・右からボーカル、と左右で違う音

音楽はステレオだと臨場感が出ますが、**音声認識には左右の区別はいりません**。

何をしゃべったか分かればいいので、データが軽くて済むモノラルを使います。

## record って何？

ここまで分かると、`record` の役割がすっきり見えてきます。

`record` は、**マイクを掴んで、さっきの設定どおりのPCMデータを取り出してくれる録音パッケージ**です。

やってくれることは、ざっくり言うとこれだけです。

- マイクの利用許可を確認する
- 指定した設定（16kHz / PCM16 / モノラル）で録音を始める
- マイクの音をPCMデータとして渡してくれる
- 録音を止める

逆に言うと、`record` は**音を取ること以外はしません**。

音を文字にするのも、話している部分を切り出すのも、`record` の仕事ではありません。

### 2つの使い方：ファイル保存とストリーム

`record` には大きく2つの使い方があります。

| 使い方 | 音の行き先 | 用途 |
|---|---|---|
| ファイルに保存 | 端末に録音ファイルが残る | ボイスレコーダー、あとで聴く |
| ストリームで受け取る | データが流れてくるだけ（保存しない） | リアルタイム処理 |

音声認識アプリでは、後者の**ストリーム**を使います。

マイクの音を「川のように流れてくるデータ」として受け取り、その場で処理して、音そのものは保存しません。

イメージとしてはこうです。

```text
マイク → record → PCMが少しずつ流れてくる → その場で処理 → 音は捨てる
```

最終的に端末に残すのは、文字にした**メモ**だけです。

録音ファイルを溜め込まないので、容量にもプライバシーにも優しい、という設計になります。

### 設定はこんなふうに渡す

コードの細かい話は省きますが、雰囲気だけ載せておきます。

さっきの「16kHz / PCM16 / モノラル」を、そのまま設定として渡すだけです。

```dart
const config = RecordConfig(
  encoder: AudioEncoder.pcm16bits, // PCM16（1サンプル2バイト）
  sampleRate: 16000,               // 16kHz（1秒に16000回）
  numChannels: 1,                  // モノラル（1本）
);
```

ここまで読んでいれば、この3行が「何を指定しているのか」はもう分かるはずです。

前提知識さえあれば、録音の設定は驚くほどシンプルです。

## record と sherpa-onnx の役割分担

僕は最初、「録音と音声認識って、1つのライブラリでまとめてやるのかな？」と思っていました。

でも実際は、役割がきれいに分かれています。

| パッケージ | 役割 |
|---|---|
| `record` | マイクの音をPCMデータとして**取り出す** |
| `sherpa-onnx` | そのPCMデータを**文字にする**（VADで区切るのも担当） |

バケツリレーのような関係です。

```text
record（音を取る）→ sherpa-onnx（区切って、文字にする）
```

`record` が取り出したPCMを、`sherpa-onnx` に渡す。

それぞれが1つの仕事だけを担当するので、構成がシンプルになります。

「録音は `record`、認識は `sherpa-onnx`」と覚えておけば、頭の中が整理しやすいです。

## 実際に使うときに気になりそうなこと

`record` 自体はシンプルですが、アプリに組み込むとなると、いくつか考えることがあります。

### マイクの権限

当然ですが、マイクを使うにはユーザーの許可が必要です。

iOS なら「マイクを何に使うか」の説明文、Android なら録音の権限が必要になります。

`record` は許可の確認もしてくれるので、そこに合わせてアプリ側の権限フローを用意します。

### iOS と Android の違い

`record` は内部で iOS / Android それぞれのネイティブな録音機能を使っています。

基本的な使い方は共通ですが、対応する設定やOSバージョンに細かい差があるので、両方の実機で確認しておくと安心です。

### データはどこに行くのか

ストリームで使う場合、流れてくるPCMは**メモリの上を通り過ぎるだけ**です。

特に何もしなければ、処理が終わった音のデータはそのまま捨てられます（端末には残りません）。

「録音ファイルが勝手に溜まっていく」みたいなことは起きないので、その点は安心です。

逆に「あとで聴けるように残したい」場合は、自分でファイル保存の使い方を選ぶ必要があります。

## まとめ

`record` は、名前のとおり「録音」のためのパッケージですが、その中身を理解するには、

**そもそも音がどうやってデータになるのか**

を先に知っておくと、一気に分かりやすくなります。

一言でいうなら、`record` は、

**マイクの音を、PCMという生データとして取り出すための入り口**

です。

ポイントを整理すると、こんな感じです。

- 音は「波」で、それを一定間隔で測って数字にしたものが **PCM**
- 録音では **サンプリングレート / ビット深度 / チャンネル** を決める
- 音声認識アプリでは **16kHz / PCM16 / モノラル** を使うことが多い
- `record` は音を**取るだけ**。文字にするのは `sherpa-onnx` の仕事
- ストリームで使えば、音は保存せずその場で処理して捨てられる

`record` 単体では「音を文字にする」ことはできませんが、その手前の「マイクの音をきちんとデータにする」という、地味だけど一番大事な土台を担っています。

音声まわりのアプリを作るなら、まずはこの「録音とは何をしているのか」を押さえておくと、その先のVADや音声認識の話もぐっと分かりやすくなると思います。
