# Step 13: 失敗からの自動復帰（自動再試行）

## ひとことで言うと

エンジン準備が失敗したら、**ボタンを出さずに自動で再試行し続ける**。表示は "Retrying" のひとことだけ。
再試行が続けて失敗したら **"Try restarting"（再起動の提案）に表示を切り替え、裏では遅いペースで再試行を続ける**。

---

## なぜこの形なのか

- このアプリは準備が終わらない限り何もできない。失敗時にユーザーが取れる行動はほぼ「再試行」だけなので、ボタンを押させる意味がない。自動なら放置でも復帰する。
- ただし再試行で直らない失敗（ビルド不良・環境の引っかかり等）もある。無限に "Retrying" を出し続けると「固まったアプリ」と見分けがつかないため、**連続5回失敗したら再起動の提案に切り替える**。再起動はネイティブ状態・メモリ・Play まわりが全部やり直しになる万能リセットで、ユーザーが必ず理解できる唯一の提案。
- 切り替え後も裏で再試行を続けるのは、ネットワーク断のような「待てば直る失敗」を拾うため。電波が戻れば再起動なしで勝手に復帰する（"Try restarting" は指示ではなく提案）。

---

## やること

### 1. 再試行ループを `SttEngineNotifier` に足す

- 再試行の入口とバックオフのタイマーは `SttEngineNotifier`（`lib/providers/stt_providers.dart`）に置く。
- ゲートの分岐（`RootView`）も待ち画面（`PreparationGatePage`）もライフサイクルを持たない作りのため、失敗を最初に知る provider が自分で再試行を予約する。ユーザー操作は挟まない。

### 2. 再試行の中身は2つで1組

1. DL起因の失敗（未開始・DL失敗）なら `SttModelProvisioner.requestModelDownload()` で再取得を頼む。
2. エンジン準備（`build()` の処理）をやり直す。

### 3. バックオフ付きで自動リトライ

- 失敗のたびに間隔を広げる（**2秒 → 4秒 → 8秒 → 16秒 → 30秒。以降は上限の30秒間隔**）。
  - 指数的であること自体に強い根拠はない（自前サーバーは無く、保護対象がない）。間隔を広げる実益は、長引く失敗（圏外等）での電池・無駄な処理の抑制と、表示のチラつき抑制。
- 成功したら間隔をリセットする。

### 4. 続けて失敗したら "Try restarting" に切り替える

- **連続5回**失敗したら、表示を "Retrying" → "Try restarting" に切り替える。
  - 時間の基準は別に設けない。バックオフの累計で5回時点が約1分になるため、回数だけで実質同等になる。
- 切り替え後も再試行は止めない（上限の**30秒間隔**のまま続ける）。成功したらそのままリスニングへ進む。
- 閾値は控えめにしすぎない。一番よく起きる失敗はネットワーク断で、これは再起動では直らず待てば直る。数回の失敗で再起動を勧めると逆効果。

### 5. 待ち画面の判定順を1か所直す

- 表示の文言・スライド演出は実装済み（新しいUIは作らない）。"Try restarting" のフェーズを `PreparationPhase` に足す。
- ただし `PreparationGatePage` の `_resolveStatus` は**エンジンのエラーを最優先**で見ている。AsyncNotifier の再実行中は直前のエラーが保持される（`hasError` が true のまま）ため、このままだとDLが再開しても "Retrying" に固定される。再実行中（`isLoading`）はDL状態を優先する等、判定順を直す。

---

## やらないこと

- 再試行ボタン・エラーダイアログ・詳細なエラー説明の表示（短いひとことだけ）
- 失敗原因の分類（直る系/直らない系の判定）。エラーコードから確実に分かるのは一部だけなので、分類せず「閾値まで失敗が生き残ったら再起動を提案」に一本化する

---

## 完了の目安

- 機内モード等でDLを失敗させたあと通信を戻すと、**操作なしで** Retrying → Downloading n% → Almost there → リスニングへ復帰する。
- 失敗が続くと "Retrying" → "Try restarting" に切り替わり、固まったように見えない。
- "Try restarting" 表示後も、通信を戻せば操作なしで復帰する（裏の再試行が続いている）。
- リトライの間隔が広がっていく（バックオフが効いている）。
- `flutter analyze` が通る。

---

## 実装メモ

- **Riverpod 3 は失敗した provider を既定で自動再試行する**（0.2〜6.4秒の指数バックオフ）。自前のバックオフと二重に走り、実際の再試行ペースが Riverpod 側に支配されてしまうため、`sttEngineProvider` は `retry: (retryCount, error) => null` で無効化する（実機検証でログの間隔ズレとして発覚）。
- タイマーは `SttEngineNotifier` のライフサイクルに紐付け、dispose 時・成功時に確実に止める。
- 「DL未開始」も失敗として届く（`_waitForModelDownload` が downloading 以外を即エラーにする設計）。この場合も同じ経路（再取得 → 準備やり直し）で復帰する。
- `requestModelDownload()` 直後はDL状態がまだ notStarted のことがある。すぐ準備をやり直すと即エラーになりバックオフが空回りするので、downloading への遷移を少し待ってからやり直す。
