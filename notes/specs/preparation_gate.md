# 準備ゲート仕様

## 概要
リスニングに入る**直前の1か所**で、文字化エンジン（オンデバイス STT）の準備が完了しているかを判定するゲートです。

- **準備済みなら素通り**: 待ち画面は表示されず、そのままリスニング画面へ進みます。
- **未完了なら足止め**: 待ち画面で現在の状況を表示し、準備が完了したら自動でリスニング画面へ進みます。ユーザー操作は不要です。

エンジンの準備（メモリ読み込み・実測約1.5秒）は起動と同時に裏で始まっており、スプラッシュ・権限フローの間に終わるため、**通常起動は素通りが基本**です。足止めが起きるのは主に Android 初回起動（モデル 約625MB の自動ダウンロード中）です。

## 画面遷移上の位置

```text
App Start（エンジン準備を裏で開始）
→ Splash
→ Permission
→ 【 準備ゲート 】 ── 準備済み ──→ Listening Screen
        └─ 未完了: 待ち画面で足止め → 完了したら自動で進む
```

待たせる処理はこの1か所に集約されており、スプラッシュや権限フローに足止めはありません。

## 待ち画面の表示状態（UI State）

待ち画面はスプラッシュと同じ「中央の1行テキスト」デザインです。いま何をしているかを短い英語ひとことで伝え、**進捗バーは置きません**。

| フェーズ (Phase) | 表示テキスト | 状況 |
| --- | --- | --- |
| gettingReady | `Getting ready` | ダウンロード開始前・DL状態の取得前 |
| downloading | `Downloading 37%` | モデルの自動ダウンロードが進行中 |
| almostThere | `Almost there` | ダウンロード完了、エンジンのメモリ読み込み中 |
| retrying | `Retrying` | 準備に失敗し、自動再試行を待っている |
| tryRestarting | `Try restarting` | 連続失敗が閾値を超え、再起動を提案している |

- テキストの切り替えは**フェーズが変わったときのみ**スライド演出で行います。ダウンロード中の `%` は数字だけその場で更新します（毎%スライドはさせない）。
- `Try restarting` 以外のフェーズは「処理中」を示すドットアニメーション付き、`Try restarting` はユーザーへの依頼のため動きを付けません。

## フェーズの判定ルール

エンジンの準備状態と、モデルの自動ダウンロード状態（Android の fast-follow）を併読して判定します。

1. **エンジンが失敗中**（再試行の実行中は除く）→ `retrying`。ただし再起動提案フラグが立っていれば `tryRestarting`。
2. **DL状態が未取得** → `gettingReady`。
3. **DL状態ごとに出し分け**: 未開始 → `gettingReady` ／ DL中 → `downloading` ／ 完了 → `almostThere` ／ 失敗 → `retrying` または `tryRestarting`（1. と同じ判定）。

iOS のように自動ダウンロードが無い環境では、DL状態は常に「完了」扱いになるため、足止めが起きた場合の表示は `Almost there` のみです。

## 失敗からの自動復帰（自動再試行）

準備が失敗しても、**ボタンなし・ユーザー操作ゼロ**で自動再試行し続けます。再試行ループはエンジン側（provider）が持ち、待ち画面は状態を表示するだけです。

- **バックオフ**: 再試行の間隔は失敗のたびに 2 → 4 → 8 → 16 → 30秒 と広がり、以降は30秒間隔を維持します。
- **DL起因の失敗**: モデルが未開始・DL失敗で止まっている場合は、再取得を依頼してから準備をやり直します。
- **再起動の提案**: 連続5回失敗したら表示を `Try restarting` に切り替えます。無限の `Retrying` だと固まったアプリと見分けがつかないためです。**切り替え後も裏では30秒間隔で再試行を続ける**ので、待てば直る失敗（通信断など）は再起動なしで復帰します。
- **復帰**: 準備が成功すると失敗カウントと再起動提案はリセットされ、自動でリスニング画面へ進みます。失敗原因の分類（直る系/直らない系）はしません。
