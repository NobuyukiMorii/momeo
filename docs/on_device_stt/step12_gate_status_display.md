# Step 12: 待ち画面のステータス表示（スプラッシュ風）

## ひとことで言うと

Step 11 の仮の待ち画面を、スプラッシュと同じ「**中央の1行テキストが流れていく**」デザインに仕上げる。
いま何をしているかは短い英語ひとことで示し、DLの進み具合は "Downloading 37%" の**数字**で伝える。
**進捗バーは置かない**。

---

## デザインの方針

お手本は `lib/pages/splash_page.dart`。待ち画面もこれに寄せる。

- 部品はスプラッシュと同じ: `IntroSettingLayout` ＋ `AppTextStyles.headline` の中央1行テキストのみ。
- フェーズが進んだら、`AnimatedTextSequence` と同じスライド演出（今のテキストが左へ退場、次のテキストが右から登場）で次の文言に切り替わる。
- **進捗バーを置かない理由**: momeo の起動体験は「短い言葉が流れていく」で統一されており、バーだけが異物になる。進捗は数値がどんどん増えることで十分伝わる。

---

## 文言（確定）

| フェーズ | いつ | 表示 |
|---|---|---|
| DL開始待ち | Android 初回、Play のDLがまだ始まる前 | **Getting ready** |
| DL中 | Android 初回 | **Downloading 37%**（数字部分だけその場で更新） |
| メモリに読み込み中 | 準備が間に合わなかったとき | **Almost there** |
| 失敗 → 再試行中 | 失敗時（自動再試行の発動は Step 13） | **Retrying** |

- トーンはスプラッシュの `'Open. Speak. Saved.'` `'Auto-start'` に合わせる: **短い言い切り。「…」や説明文は付けない**。
- 「止まっていない」ことが伝われば十分。詳しい説明はしない。
- 完了時の文言は無い（完了したら画面ごとリスニングへ遷移するため）。

Android 初回の流れの例:

```
Getting ready → Downloading 0% … 100% → Almost there → （リスニングへ遷移）
```

---

## 時間駆動と状態駆動 — 新しいウィジェットが要る

既存の `AnimatedTextSequence` は「1.5秒ごとに次のテキストへ」という**時間駆動**。
一方この待ち画面は「準備状態が変わったら次のテキストへ」という**状態駆動**で、切り替わるタイミングを事前に決められない。

そこで、**「外から渡されたテキストが変わったら、スライドで切り替わる」状態駆動版のウィジェット**を作る。
見た目・動き（スライドの方向・時間）は既存の `AnimatedTextSequence` と同じに揃える。

### 共通化の方針: 描画部分だけ共有し、トリガーは作り分ける

丸ごと別物として作ると「デザインは同じなのに実装が2つ」になり、片方だけ直して見た目がズレる保守リスクがある。かといって、1つのウィジェットに時間駆動／状態駆動のモード分岐を持たせると、「次はいつ切り替わるか」という本質的に別のロジックが混ざって読みにくくなる。

そこで、共有するのは**スライド演出の描画部分だけ**にする。

- `AnimatedTextSequence` のうち「現在テキストが左へ退場・次テキストが右から登場」を描く部分は、トリガーが時間でも状態でも完全に同じ。ここを「現在テキスト・次テキスト・進行度を受け取って描くだけ」の共通ウィジェットとして切り出す。
- その上に薄いラッパーを2つ置く:
  - **時間駆動**（既存 `AnimatedTextSequence`）: 一定時間ごとに次のテキストへ進める。スプラッシュ用。外から見た挙動は変えない。
  - **状態駆動**（このステップで新設）: 渡されたテキストの変化を検知してスライドを起動する。待ち画面用。
- 切り出しの粒度（`AnimationController` の生成・破棄まで共通側に含めるか等）は実装時に詰める。

動きの原則:

- **スライドはフェーズの切り替わり時だけ**（Getting ready → Downloading、Downloading → Almost there など）。
- **DL% の数字はその場で更新**する（毎パーセントでスライドさせるとチカチカするため）。

---

## やること

### 1. 状態駆動スライドテキストのウィジェットを作る

- まず `AnimatedTextSequence` からスライド演出の描画部分を共通ウィジェットに切り出す（上記「共通化の方針」）。
- その上に状態駆動版のラッパーを新設する。既存のスプラッシュ（時間駆動）の見た目・挙動は変えない。

### 2. DL状態を画面から読めるようにする

- 「DL中」と「メモリ読み込み中」は、エンジン（`sttEngineProvider`）から見るとどちらも「準備中」。出し分けるには **Step 8 が公開するDL状態（進捗つき）を併せて見る**。
- DL状態を画面から読みやすくする小さな配線（provider）が要るなら、このステップで足す。

### 3. フェーズ → 文言のマッピング

- 上の文言表のとおりに出し分ける。"Retrying" は文言の定義だけ先にしておく（実際に出るのは Step 13 以降）。

---

## このステップでやらないこと

- ゲートの分岐そのもの（素通り／足止め） → Step 11 で実装済み
- 失敗からの自動復帰（再試行の実行） → Step 13
- DL状態の公開そのもの → Step 8

---

## 完了の目安

- Android 初回（bundletool の模擬配信）で、Getting ready → Downloading n% → Almost there と流れてリスニングへ繋がる。
- DL% の数字がなめらかに増え、スライド演出はフェーズ切替時だけ起きる。
- 待ち画面の見た目がスプラッシュと揃っている（並べて見て違和感がない）。
- 描画部分の切り出し後も、スプラッシュ（時間駆動）の見た目・挙動が変わっていない。
- `flutter analyze` が通る。
