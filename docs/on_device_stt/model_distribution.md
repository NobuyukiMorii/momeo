# モデル配布の詳細（同梱方式の実装と申請）

## この文書について

- **目的**: 625MB のオンデバイス STT モデルを、ユーザーの端末にどう届けるか（配布方式）を詳しく説明する。実装計画の全体像は `outline.md` にあり、本書はそのうち「モデル配布」だけを深掘りした補足資料。
- **前提知識**: アプリ配布やストア申請の経験がなくても読めるように、用語からかみ砕いて説明する。
- **結論（先に）**: 配布方式は **「道1：アプリに同梱」を基本**とする。ただし **iOS と Android で実装方法も申請設定も異なる**（理由は後述する「Android の 200MB 制限」）。さらに **Android の NeMo（625MB）は、当初の install-time から PAD の `fast-follow`（インストール直後に Play が自動ダウンロード）へ変更**した。コピー不要・ディスク半減・実装が軽くなるため。判断の経緯は [../research/on_device_stt/model_delivery_decision_for_beginners.md](../research/on_device_stt/model_delivery_decision_for_beginners.md) を参照。

---

## 0. まず用語（最初に押さえると以降が読みやすい）

| 用語 | かみ砕いた意味 |
|---|---|
| **モデル** | 音声を文字にする「脳」にあたるファイル。今回は `model.int8.onnx`（約625MB）＋ `tokens.txt`。これを端末に置かないと文字化できない |
| **同梱（バンドル）** | アプリの中にファイルを入れて一緒に配ること。ストアからインストールした時点で端末に入っている状態 |
| **アセット** | アプリに同梱するファイルのこと（画像・音声・モデルなど）。Flutter では `pubspec.yaml` の `assets:` に書いて含める |
| **AAB / APK**（Android） | AAB はストアに提出する「素材一式」。Google Play がそこから端末ごとに最適な APK（実際にインストールされる形）を作って配る |
| **IPA**（iOS） | iOS アプリをストアに提出・配布するときのパッケージ形式 |
| **PAD（Play Asset Delivery）** | Android で「大きなファイルを配るための公式の仕組み」。普通のアセットでは入りきらない大容量を扱うために使う |
| **install-time / fast-follow** | PAD の配信モード。install-time は「インストール時に同梱」、fast-follow は「インストール**直後**に Play が自動DL」。本プロジェクトは NeMo に **fast-follow** を使う |

---

## 1. 配布の3つの道（全体像）

625MB のモデルを端末に届ける方法は、大きく3通りある。

| 方式 | 概要 | 長所 | 短所 |
|---|---|---|---|
| **道1 同梱（基本採用）** | ストアインストール時点（または直後）でモデルが入る | 自前サーバ不要・Play 任せ。iOS は完全に同梱で即オフライン | Android は追加設定が要る。NeMo は install-time だと重いので fast-follow に変更（後述） |
| 道2 ストア配信 | ストアがホストし、インストール後に自動/任意でDL | 初回インストールは軽い。ホスティングはストア任せ | 初回に「取得待ち」が出る。設定はやや複雑 |
| 道3 自前DL | Hugging Face や独自サーバから初回DL | モデルだけ後から差し替えやすい | DL画面・Wi-Fi限定・再開・整合性チェックを全部自作。サーバの安定運用も自前 |

→ 個人開発で「自前サーバを持たず、Play 任せで配る」ことを優先し、**道1（同梱）を基本**とする。ただし Android の NeMo だけは、後述のとおり PAD の中でも **fast-follow（インストール直後に自動DL）** を使う（install-time だとコピー必須・ディスク約1.25GB・自前ネイティブが要るため）。これは実質「道2 寄り」だが、配信は Play 任せで道3の自作負担は無い。

---

## 2. なぜ iOS と Android で違うのか（最重要ポイント）

同じ「同梱」でも、**iOS と Android では“普通に入れられるサイズの上限”が違う**。これが実装差のすべての原因。

- **iOS**: アプリ本体は最大 **4GB** まで許容。625MB のモデルは**アプリ本体にそのまま同梱できる**（PAD のような特別な仕組みは不要。同梱の具体策は §3-1）。
- **Android**: Google Play は「最初に端末がダウンロードする本体部分」を **200MB まで**に制限している。625MB はこれを大きく超えるので、**普通のアセットには入れられない**。そこで大容量専用の仕組み **PAD（Play Asset Delivery）** を使う必要がある。

```text
モデル 625MB を「同梱」したい
  ├─ iOS   … 4GB枠に余裕で収まる → そのまま同梱でOK
  └─ Android … 200MB枠を超える   → PAD（大容量専用の仕組み）が必要
```

> つまり「両OSまとめて625MBを同梱する統一的な方法」は無く、**iOS と Android で別々のやり方になる**。

---

## 3. 道1（同梱）の実装：iOS と Android

### 3-1. iOS の場合

- モデルを **Xcode の「バンドルリソース」として同梱**する（Flutter の `pubspec.yaml` の `assets:` ではなく、Xcode プロジェクトに直接加える）。アプリ本体の一部としてそのまま配布される。
- **なぜ Flutter の普通のアセットにしないか**: Flutter アセットは iOS だと「実ファイルのパス」が取れず、sherpa に渡すために 625MB を書込領域へ**まるごとコピー**する羽目になる（同じ物を2つ持ち、ディスク約1.25GB）。
- **Xcode のバンドルリソースなら、アプリ内の実ファイルのパスが直接取れる**ので、**コピー不要**（ディスク約625MB・起動も速い）。iOS ネイティブに「モデルのパスを返す」数行の橋渡し（`Bundle.main.path`）を足すだけで済む。
- 625MB がアプリ本体に入る点・ストアからのDL量はどちらも同じ。違いは**インストール後にコピーするかどうかだけ**なので、コピーの要らないバンドルリソースを選ぶ（Android の fast-follow で「625MB の二重持ちを避ける」と決めたのと同じ考え方）。

### 3-2. Android の場合

- 普通のアセットは200MB制限で不可。代わりに PAD の **「fast-follow アセットパック」** という入れ物を用意する。
  - fast-follow は「**インストール直後に Play が自動でダウンロード**する大容量パック」。本体インストールには含まれないので、初回インストールは軽い。
  - ダウンロード完了後は、`getPackLocation().assetsPath()` で **展開済みフォルダの実パス**が得られ、そのまま sherpa に渡せる（**コピー不要**）。
  - これにより install-time の難点（`AssetManager` 経由でしか読めず 625MB コピー必須・ディスク約1.25GB・自前ネイティブブリッジが必要）を避けられる。判断経緯は [../research/on_device_stt/model_delivery_decision_for_beginners.md](../research/on_device_stt/model_delivery_decision_for_beginners.md)。
- プロジェクトには、このアセットパック用の小さな専用モジュールを定義し、ビルド設定（Gradle）に紐付ける**Android専用の配線**が必要になる。
- 引き換えに、**「インストール直後はまだDL中でモデルが無い」状態**が生まれる。初回起動時に進捗を見せ、揃うまで音声入力を待たせる扱いが要る（momeo の Intro/Setting オンボーディング中に背景DLが進む想定）。
- **権限**: PAD（`asset-delivery` ライブラリ）を入れると、その manifest 由来で `INTERNET`＋`ACCESS_NETWORK_STATE` がマニフェストマージで**自動的に付く**。どちらも normal permission なので**ユーザーに許可ダイアログは出ない**（マイクの `RECORD_AUDIO` とは別レイヤー）。**音声認識の実行自体はオフライン**で、ネットを使うのは初回のモデルDLだけ。

### 3-3. 実行時の共通処理（パス契約）

- sherpa-onnx は「**実ファイルのパス**」を渡さないとモデルを読めない。そこでアプリは最終的に **「決まったパスに有効なモデルがある状態」を保証**し、下流（文字化）には**実パスだけ**を渡す。これを「**パス契約**」と呼ぶ。
- パスの用意の仕方は、モデルと OS で異なる。

| 対象 | 取得元 | 実パスの作り方 |
|---|---|---|
| `silero_vad.onnx`（全OS） | Flutter アセット（`rootBundle`、実体なし） | 書込領域へ**コピー**（Step 5 で実装済み） |
| NeMo（iOS） | アプリに同梱（**Xcode バンドルリソース**） | `Bundle.main` の**実パスを直接利用（コピー不要）** |
| NeMo（Android） | fast-follow アセットパック | `getPackLocation().assetsPath()` の**実パスを直接利用（コピー不要）** |

- ポイントは、**NeMo は iOS・Android ともコピー不要**になったこと（625MB の二重持ちを両OSで避けられる）。実際にコピーするのは小さい `silero_vad.onnx`（約0.6MB）だけ。
- 取得元の違い（アセット／パック／開発時の事前配置）は、このパス契約の内側に閉じ込める。

## 4. ストア申請時に必要な設定

### 4-1. iOS（App Store Connect）

- **モデル固有の特別設定はほぼ不要**。モデルはアプリ本体の一部なので、通常どおりアップロードするだけ。
- 影響するのは「アプリサイズが大きくなる（~700MB級）」点くらい。
- セルラー回線で200MBを超えるアプリは「Wi-Fi 推奨の確認ダイアログ」が出るが、これは**自動**で、こちらの設定項目はない。
- 申請時の定番項目（暗号化の輸出コンプライアンスの質問など）は、モデルとは無関係に従来どおり答える。

### 4-2. Android（Google Play Console）

- **AAB に fast-follow アセットパックを含めて**アップロードする。
- **自前ホスティングは不要**（Google が配信してくれる）。
- fast-follow パックは**インストール直後に Play が自動DL**する。初回インストール自体は軽く、モデルは少し遅れて届く。
- ポイント：200MB を超える配布になるため、**PAD を組んでから提出する**のが Android 側の実体。これが「Androidだけ申請前に整える設定」。

| | iOS | Android |
|---|---|---|
| 申請前の準備 | 特別な準備ほぼなし（大きいバイナリをそのまま提出） | AAB に fast-follow アセットパックを組み込む |
| ホスティング | 不要 | 不要（Google 配信） |
| 大きさの扱い | 4GB枠内なら自由 | fast-follow パックで200MB枠外として配布（インストール直後に自動DL） |

## 5. 開発中（`flutter run`）の注意

- **iOS**: 同梱アセットは開発ビルドでもそのまま動く。特別な手間はない。
- **Android**: PAD（fast-follow）はストア配信が前提のため、ローカル実行ではアセットパックがそのままでは届かない。開発時は `adb push` でモデルを暫定配置するか、bundletool でストア配信を模擬する一手間が要る（**iOS より検証が面倒**）。具体策は [step06_provision_models.md](step06_provision_models.md)（`adb push` 事前配置）と [step08_android_fast_follow.md](step08_android_fast_follow.md)（`bundletool` で fast-follow 経路確認）を参照。

## 6. 補足

### 6-1. なぜ「圧縮して小さく」が効かないか

- 採用モデルは**すでに int8 量子化済み**（パラメータを最小限のサイズまで削った状態）。これ以上の標準的な量子化は提供されておらず、自前変換は精度劣化と手間が大きい。
- zip 等の汎用圧縮も、量子化済みデータは縮みにくく、しかも**ストア配信時に既に圧縮**されている（サイズ判定も圧縮後）。自前zipの上乗せ効果はわずかで、200MBには届かない。
- → サイズを本気で小さくする唯一の手は「**より小さいモデルに替える**」こと（例: k2 Zipformer 159M）。ただし精度・句読点を犠牲にするため、本プロジェクトは NeMo を維持する。

### 6-2. 将来サイズが200MB以下になれば差は消える

- もし軽量モデルに替えてアプリ全体が200MB以下に収まれば、Android も普通のアセット同梱で済み、PAD は不要になる。そのとき iOS/Android の実装差はほぼ消える。
