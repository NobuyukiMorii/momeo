# Step 6: モデルを「読める置き場所」に用意する（配置とパス契約の土台）

## ひとことで言うと

音声を文字にするには、2つのモデルファイルを**端末の中に置いておく**必要がある。このステップは、その**置き場所を用意する**作業。

- **NeMo（約625MB）** … 日本語を文字に変換する本体
- **`silero_vad.onnx`（約0.6MB）** … 「話し始め・話し終わり」を見つける係（Step 5 で配置済み）

大事な前提が1つ。文字化エンジン（`sherpa`）は、**ファイルの「住所」（端末のどこに置いてあるか＝パス）を教えてもらわないとモデルを読めない**。
だからこのステップのゴールは、ただ置くことではなく、「**決まった住所に、ちゃんと読めるモデルが必ずある**」状態を作ること。

なお、ここでやるのは「置く」まで。実際に「メモリに広げて使える状態にする」のは Step 9、「音声を文字にする」のは Step 7。

> **このステップは土台に絞る。** Android にモデルを**本番で**届ける仕組み（`fast-follow`）は重たいので Step 8 に切り出した。
> ここでの Android は、開発機に手でモデルを置いて（`adb push`）、先の工程を進められる状態にするところまで。
> 本番配信・初回ダウンロードの待ち画面・必要な権限の話は [step08_android_fast_follow.md](step08_android_fast_follow.md) にまとめてある。

---

## このステップの目的

音声メモができるまでの流れの中で、ここは「**文字化の材料（モデル）を先に用意しておく**」係。

```
マイク → 録音 → 区切り → 文字化 → メモに保存
                          ↑
              文字化に使うモデルファイルを、
              先に端末へ置いておく（このステップ）
```

### 「置く」と「読み込む」は別のこと

言葉が似ているので分けておく。このステップは**置く**だけ。

| | 何をする | いつ | 担当 |
|---|---|---|---|
| **置く（このステップ）** | モデルを端末の中に、読める形で用意する | インストール後に **1回だけ** | Step 6 |
| 読み込む | 置いたモデルを**メモリに広げて、使える状態**にする | アプリを開く **たびに** | Step 7 で作り、Step 9 で起動時に組み込む |

「読み込む」と、その数秒の待ちをスプラッシュ画面で隠す工夫は Step 9 の担当。

---

## モデルごとの「置き方」（住所が取れるかで変わる）

「置く」とは、ただコピーすることではなく、**最後に文字化エンジンへ渡せる「住所」を用意すること**。
置き方はモデルと OS で違い、下の3パターンになる。

| 対象 | どこから来るか | 住所（パス）の作り方 |
|---|---|---|
| `silero_vad.onnx`（iOS/Android 共通） | Flutter の普通の同梱（アセット）。ただし**そのままでは住所が無い** | 端末の書き込みできる場所へ**コピー**して住所を作る（Step 5 で実装し、本ステップで窓口に集約） |
| NeMo（iOS） | Xcode（iOS 用の作り込み道具）に**直接持たせる**「バンドルリソース」 | アプリの中の**本物の住所がそのまま取れる**のでコピー不要（`Bundle.main` から取得） |
| NeMo（Android） | **今は**開発機に手で置く（`adb push`）。**本番**の自動配信は Step 8 | 置いた場所の**住所をそのまま使う**（本番では Step 8 で住所の作り方を差し替え） |

`silero_vad.onnx` は Step 5 で置き済み。**このステップの山は、625MB の NeMo の住所を用意すること**。
iOS は本物の同梱（バンドルリソース）でこのステップ内に完成させ、Android は本番の届け方を Step 8 に回して、まずは手置きで「住所のしくみ」が動く状態にする。

> **なぜ iOS は「バンドルリソース」なのか。** Flutter の普通の同梱（アセット）にすると、iOS では**ファイルの本物の住所が分からない**。すると使う前に 625MB を別の場所へ**まるごとコピー**することになり、同じものを2つ持つ（約1.25GB）。Xcode のバンドルリソースにすれば**住所がそのまま取れて**コピー不要（約625MB のまま・起動も速い）。どちらもアプリ本体に 625MB が入る点・ストアからのダウンロード量は同じで、違いは「あとでコピーするかどうか」だけ。だからコピーの要らないバンドルリソースを選ぶ。実装は iOS 側に「モデルの住所を返す」数行を足すだけ。

> 配り方のさらに詳しい話（iOS と Android の違い・`PAD`・ストア申請）は [model_distribution.md](model_distribution.md) を参照。

## パス契約 ＝「決まった住所に必ずモデルがある」という約束

このステップの中心になる考え方。

**パス契約**とは、「**決まった住所（パス）に、いつでも読める本物のモデルが必ず置いてある**」とアプリが保証する約束のこと。

- **用意する側（このステップ）**：決まった住所に、ちゃんと読めるモデルを必ず置く。
- **使う側（Step 7 の文字化）**：その住所を受け取って読むだけ。モデルが**どうやって届いたか**（iOS の同梱／Android のダウンロード／開発中の手置き）は**気にしない**。

たとえるなら**宅配ボックス**。配達する側は「この住所に必ず本物を入れておく」と約束し、受け取る側は「その住所を見るだけ」。
届け方が違っても受け取り口は1つに揃うので、後で Android の届け方を変えても（Step 8）、使う側のコードは変わらずに済む。

実装は、「住所を返す」役だけを担う窓口 `SttModelProvisioner`（`lib/stt/stt_model_provisioner.dart`）に1か所でまとめる。届け方の違いは、すべてこの窓口の**中だけ**に隠す。silero・NeMo・tokens の3ファイルとも、この窓口を通して住所を受け取る。

NeMo（625MB）は大きすぎてリポジトリに入れていない（`gitignore`）。
そのため「**どこから取って・どこに置くか**」を計画に書いておかないと、別の環境にコピーを持ってきたときに「モデルが無くてビルドできない」状態になる。下表をこのステップの正とする。

### 取得元

| 項目 | 内容 |
|---|---|
| 取得元（メイン） | `https://huggingface.co/csukuangfj/sherpa-onnx-nemo-parakeet-tdt_ctc-0.6b-ja-35000-int8`（取得スクリプトは `resolve/main/` ではなく**特定コミットに固定**して取得する。理由は後述の「自動化スクリプト」） |
| フォールバック | 同じ作者の GitHub Releases（`github.com/k2-fsa/sherpa-onnx/releases`）からも sherpa-onnx 用モデルを取得できる |
| 取得元の素性 | 上記 `csukuangfj` は、**このアプリで使う sherpa-onnx の主要メンテナ本人**（Fangjun Kuang）の公式アカウント。素性不明の第三者ではない |

> Hugging Face は機械学習モデル配布の標準プラットフォーム。「HF だから安全」ではなく「**誰のアカウントの配布物か**」で判断するのが正しく、今回は**ライブラリ作者本人の公式配布**なので入手元としては最上位クラスの信頼度。

### 取得するファイル

| ファイル | サイズ（バイト） | 置き場所 |
|---|---|---|
| `model.int8.onnx` | **655,542,604** | iOS: `ios/Runner/Models/`／Android: 本番はアセットパック（Step 8）、開発中は端末の内部ストレージへ手置き（→「開発中の Android の手置き」） |
| `tokens.txt` | **28,557** | 同上 |

#### なぜ2ファイル必要か（`tokens.txt` の役割）

音声認識モデル（`.onnx`）は、実は**文字を直接は出力しない**。出てくるのは「**番号の列**」（例: `[5, 102, 8, ...]`）で、これだけでは意味が分からない。

`tokens.txt` は、その「**番号 → 文字**」の対応表（例: `5 → こ`、`102 → んにちは`）。番号を文字に直して初めて文章になる。

- 役割分担: `.onnx` は「音 → 番号」（巨大・625MB）、`tokens.txt` は「番号 → 文字」（小さい・28KB）。
- **この2つは必ずペア**。番号の振り方はモデルごとに違うため、別バージョンの表を混ぜると文字化けする。だから**同じバージョンで揃えて取得**する（取得スクリプトもコミットを固定して両方を1バージョンに揃えている）。

- **リポジトリには入れない**（`gitignore`）。取得・配置は下記スクリプトで再現する。

### 整合性チェックは「バイト数一致」だけにする（ハッシュは不採用）

ダウンロードは途中で切れても、半端なファイルが「成功」として残ることがある（HF はリダイレクトが挟まるため）。
そこで配置後に「**ファイルがあるか＋サイズがちょうど上表のバイト数か**」を確認し、これを合格基準にする。

- 途中切れ・半端DLは長さがズレるので、**サイズ一致の確認だけでほぼ全部弾ける**。
- **SHA-256 などのハッシュ検証はあえて行わない**。ハッシュが勝てるのは「サイズはピッタリ同じなのに中身だけ壊れている」という極めて稀なケースだけで、それも起きれば結局**読み込み時に失敗**して気づける。個人開発でハッシュを記録・更新し続ける保守コストに見合わないため、シンプルさを優先して採らない。
- 改ざん対策としてのハッシュも、入手元が**作者本人の公式配布**である以上、個人開発で警戒する優先度は低い。

> **期待バイト数は2か所で使う。** シェル側の共通定数 `scripts/lib/nemo_model_constants.sh`（取得・配置スクリプトが DL直後の検品などに使う）と、窓口 `lib/stt/stt_model_provisioner.dart`（端末で読む直前の最終確認）。チェックする場所・タイミングが違うので両方に意味がある（出荷時の検品と受取時の検品）。同じ数字が2か所に出るが、**1か所に集約する仕組み（マニフェスト等）は個人開発では割に合わない**ため作らない。代わりに**両ファイルに相互参照コメント**を置き、「更新時は両方直す」とだけ気づけるようにする。

### 取得・配置・登録を自動化するスクリプト

上記（取得元・ファイル・整合性チェック）を毎回手でやるのは大変で間違えやすい。そこで**3本のスクリプト**にまとめ、誰の環境でも同じ結果になるようにする。

**1. `scripts/download_nemo_model.sh`（取得）**

- 取得元を**特定コミットに固定**して `model.int8.onnx` / `tokens.txt` をダウンロードする。
  - `main`（最新を指す動く矢印）のままだと、作者がファイルを差し替えたときに中身が変わり、バイト数チェックが（壊れていないのに）失敗してしまう。固定すれば中身が変わらず、バイト数も永久に正しいまま・誰の環境でも同じファイルが手に入る。**更新したいとき**はスクリプト内の固定コミットと期待バイト数を書き換える。
- ダウンロード直後に**バイト数一致**を確認する（前掲の整合性チェック）。途中で切れたら止まり、再実行すれば続きから落とす。
- `.dev_models/`（Git 無視）に置く。配置は行わない（次項へ分離）。

**2. `scripts/place_ios_models.sh`（iOS への配置）**

- `.dev_models/` から iOS の `ios/Runner/Models/` へ**コピー**する（同じサイズなら何もしない）。
- デバイスIDを渡すと、それが Android 端末のときは何もしない（Android の手置きは `place_android_device_models.sh` の担当）。

**3. `scripts/register_ios_files.rb`（iOS の Xcode へ登録）**

- iOS の Xcode に「手で足したファイル」を、それぞれ正しいビルドフェーズに登録する（`project.pbxproj` を書き換える）。手で Xcode の GUI を操作する代わりに、`xcodeproj`（CocoaPods に同梱）で安全・確実に行う。
  - モデル2ファイル（`model.int8.onnx` / `tokens.txt`）→ **Copy Bundle Resources**（同梱する資産）
  - ネイティブ Swift（`SttModelChannel.swift`、iOS の住所を返すブリッジ）→ **Compile Sources**（コンパイルする）
- **何度実行しても二重登録しない**（idempotent）。
- これは**一度きりの編集**。`project.pbxproj` は Git 追跡対象なので、書き換えてコミットすれば他の人は `git pull` するだけで登録済みになる（モデル本体のように各PCで再実行する必要はない）。

> **なぜ GUI ではなくスクリプトか。** GUI 登録は「グループ種別の選び間違い」「ターゲット指定の付け忘れ」など間違えやすい。スクリプトなら毎回確実で、`git diff` で結果をレビューでき、やり直しも安全。考え方は取得スクリプトと同じ（手順を人の記憶に頼らず、コードに残して再現する）。

## やること

### 1. モデルを端末に置く／同梱する

- **`silero_vad.onnx`**：アセット同梱は Step 5 のまま（普通の同梱）。端末へコピーする処理の置き場所は次項「2.」で窓口に集約する。
- **NeMo（iOS）**：`download_nemo_model.sh` で取得し、`place_ios_models.sh` で `ios/Runner/Models/` に配置し、`register_ios_files.rb` で Xcode の**バンドルリソース**として登録する（→「取得・配置・登録を自動化するスクリプト」）。
- **NeMo（Android・今は）**：開発機に `adb push` で手置きする（本番の自動配信は Step 8。手順は後述）。

### 2. 「住所を返す窓口」（パス契約）を作る

- **silero**：Step 5 で VAD 画面（`stt_vad_section.dart`）に書いたコピー処理を、この窓口（`ensureSilero()`）に**集約**する。VAD 画面はこの窓口を呼ぶ形に差し替え、silero の住所解決を**1か所だけ**にする（重複ゼロ）。
- **NeMo（iOS）**：バンドルリソースの住所を `Bundle.main` から取得して返す（Dart から直接は呼べないので、`MethodChannel` のネイティブブリッジ `SttModelChannel.swift` を立て、`AppDelegate.swift` で登録して、Swift 側で住所を返す）。
- **NeMo（Android）**：今は手置きした場所の住所を返す（本番は Step 8 で中身を差し替え。窓口の外から見た形は同じ）。

### 3. ちゃんと置けたか確認する

- 置いたファイルが**壊れていない・途中で切れていない**かを確かめる。合格基準は「ファイルがあるか＋サイズが**ちょうど** `model.int8.onnx` = 655,542,604／`tokens.txt` = 28,557 バイトか」（前掲「モデルの入手元」参照）。
- とくに大きい NeMo は、コピーや配置が途中で止まる事故に備える。途中切れはサイズがズレるのでこの確認で弾ける（**ハッシュ検証は不採用**。理由は前掲）。

### 4. 開発用の確認画面（dev catalog）に表示する

- Step 5 で作った「**STT**」グループに、置き場所の状態を見るセクションを足す。
- 追加するファイル：`lib/pages/dev/catalog/sections/stt/stt_models_section.dart`
- 各モデルの**住所・サイズ・壊れていないか**を表示し、正しく置けて読めることを**目で確認**できるようにする（Android のダウンロード状態の表示は Step 8 で追加）。

## 開発中の Android の手置き（`adb push` で事前配置）

`flutter run`（PC から直接アプリを入れて動かす、開発のやり方）では、本番の自動配信（`fast-follow`）は届かない（その仕組みは Step 8 で作る）。
そこで日常の開発では、NeMo を**ビルドに含めず**、PC からコマンドで端末に一度だけ手で置く。アンインストールまで残るので毎回は不要。

> **置き場所は「内部ストレージ」にする（外部ストレージではない）。** 外部ストレージ（`Android/data/<app>/files`）へ `adb push` したファイルは、**アプリ（別UID）が読めない**。新しい Android は FUSE がアプリ自身の所有でないファイルを遮断し、`chmod` も無視されるため。そこで、アプリが確実に読める**内部ストレージ**（`getApplicationSupportDirectory()` ＝ `/data/user/0/<app>/files`、silero と同じ場所）へ置く。内部ストレージには直接 push できないので、いったん `/data/local/tmp` へ push し、`run-as`（アプリの権限）でコピーする。

事前に2つ確認する：
- `adb devices` で端末が1台出ること（実機 or エミュレータ）。
- **先にデバッグアプリを入れておく**（`flutter run`／`flutter install`）。`run-as` はインストール済みのデバッグアプリにだけ使える。

```bash
# プロジェクト直下で実行する（モデルは download_nemo_model.sh が .dev_models/ に置いている）

# 1) いったん中継地点 /data/local/tmp へ push（ここは shell が書ける）
adb push .dev_models/model.int8.onnx /data/local/tmp/
adb push .dev_models/tokens.txt      /data/local/tmp/
adb shell chmod 644 /data/local/tmp/model.int8.onnx /data/local/tmp/tokens.txt

# 2) アプリの権限(run-as)で内部ストレージへコピー（アプリが所有＝読める）
adb shell run-as jp.momeo.momeo mkdir -p files/models
adb shell run-as jp.momeo.momeo cp /data/local/tmp/model.int8.onnx files/models/
adb shell run-as jp.momeo.momeo cp /data/local/tmp/tokens.txt      files/models/
```

push したら、カタログ「STT → モデル配置」を開いて**「再取得」**を押し、NeMo・tokens が **OK（緑）** になることを確認する。

- アプリ側は、この内部ストレージ（`getApplicationSupportDirectory()` 配下の `models/`）を開発用のモデル置き場として見て、そこの住所を返す（silero と同じ置き場所に統一）。
- この内部配置は **dev 専用**。本番（Android）は Step 8 の fast-follow で `getPackLocation()` の実パスに差し替える（窓口の外から見た形は同じ）。
- 本番の自動配信の経路は、Step 8 で `bundletool` という道具を使って別途確認する。
- iOS は手置き不要。バンドルリソースがビルドにそのまま入るので、`flutter run` でも普通に動く（ビルドが少し重いだけ）。

## このステップでやらないこと

- モデルを**メモリに読み込む**ことと、その待ちをスプラッシュで隠す工夫 → Step 9

- Android の**本番の自動配信（fast-follow）まわり一式**（配信・ダウンロードの待ち画面・必要な権限） → Step 8
- リスニング画面に**つなぐ**（区切った音 → 文字 → カード） → Step 10
- ストア公開のための Android 設定 → リリース準備のときに別途
