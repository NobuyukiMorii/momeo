## 0. 30秒でVosk

- **Vosk** は、老舗の音声認識基盤 **Kaldi** をベースにした **オフライン・軽量・リアルタイム（ストリーミング）** のSTTエンジン。
- **強み**: 端末内で完結（ネット不要）／モデルが小さい（言語ごと約40〜50MB）／**40言語以上**／**話しながら逐次テキストが出る（真のストリーミング）**／CPUだけで動く。
- **弱み**: 精度は最新の end-to-end モデル（Whisper/sherpa の Zipformer など）に一歩譲る。特に**固有名詞・数字・日本語の自然さ**が弱い（momeo の実測でも分かち書き＋誤認識が目立った）。
- **立ち位置**: 「軽さ・多言語・真のストリーミング・無料」を最優先するなら今も有力。「精度最優先」なら別エンジン。

---

## 1. 仕組みのイメージ

```
🎤 マイク → PCM（生の音声・16kHz）
        │  Vosk が常時うけとる
        ▼
[ Vosk 認識器 ]  ← 話しながら少しずつデコード
   ├─ onPartial : 暫定テキスト（喋っている最中に更新される）
   └─ onResult  : 確定テキスト（無音で区切られたタイミングで出る）
```

- Vosk は**自分でマイクを掴んで連続処理**できる（`SpeechService`）。だから momeo の whisper/sherpa 構成と違い、**別途 VAD を付けなくても**動く（無音区切り＝endpointing を内部で持つ）。
- 「**partial（暫定）→ result（確定）**」の2段構え。partial は喋りながらどんどん書き換わり、ひと区切りで result に確定する。

## 2.【質問1】最初の言葉（「えーっと」「でも」）が切れる問題

「話し始めの最初のことばが入らない」現象には、主に**2つの原因**がある。

### 原因A：録音の“立ち上がり”で頭が欠ける
- マイク（Android の AudioRecord 等）は、開始してから実際に音が流れ始めるまで**わずかな立ち上がり時間**がある。「リスニング開始」を押した直後にすぐ喋ると、その**最初の数百ミリ秒が録れていない**ことがある。
- Vosk の `SpeechService` は**マイク取得を内部に隠している**ため、「発話の少し前から録っておく（先読み）」という細工が**外から入れにくい**。

### 原因B：モデルがフィラー（言い淀み）を拾いにくい
- 「えーっと」「あのー」「でも」のような**言い淀み・接続詞**は、小さいモデルや言語モデル（LM）が**重要でない音として落としやすい**。これは“切れている”のではなく“あえて拾っていない”ケース。

### 対策（やさしい順）

1. **押してから一拍おいて話す**：開始直後の立ち上がりを避けるだけで、頭欠けはかなり減る。
2. **常時リスニングにする**（押すたびに start/stop しない）：セッション開始の1回だけ立ち上がりがあり、以降の各発話の頭は欠けにくい。
3. **“先読みバッファ（pre-roll）”を自前で付ける**（しっかり対策）：
   - `SpeechService`（マイクお任せ）をやめ、**自分で録音**（`record` パッケージや VAD）し、**直近 300〜500ms を常にリングバッファに保持**しておく。
   - 発話を検知したら、**バッファに溜めた“発話直前の音”＋その後の音**を `Recognizer.acceptWaveform()` に流す。
   - これで「話し始めの頭」も含めて認識器に渡せる。※これは whisper/sherpa 構成で使っている **VAD の pre-speech padding（発話前の余白を付ける）** と同じ発想。
4. **大きいモデルに変える**：フィラーや弱い音の拾いが上がる（後述の精度調整）。

> まとめ：Vosk 本体は連続処理なので「途中の頭」を意図的には捨てない。欠けるのは主に**録音の立ち上がり**か**モデルの拾わなさ**。**先読みバッファ＋自前給餌**が一番効く対策で、フィラーまで欲しいなら**モデル強化**も併用する。

## 3.【質問2】精度の調整（速度とのトレードオフ）

精度は調整できる。効く順に：

### ① モデルサイズを上げる（最も効く）
- 各言語に **small（モバイル向け・約40〜50MB）** と **big（高精度・1GB級、メモリも大量）** がある。
- big は精度が大きく上がるが、**スマホには重い**（メモリ16GB級を要する超大型もサーバ用）。モバイルは基本 small、精度がどうしても要るなら中型を探す、が現実的。

### ② デコードパラメータ（速度↔精度のダイヤル）
Vosk（Kaldi）の中核パラメータで、モデルの `conf`（設定ファイル）にある：

| パラメータ | 意味 | 上げると |
|---|---|---|
| `beam` | 探索の広さ | 精度↑・速度↓ |
| `max-active` | 同時に保持する仮説の最大数 | 精度↑・速度↓ |
| `lattice-beam` | 確定前の候補の広さ | 精度↑・速度↓ |

- 既定はモデルごとに違う（例: 今回ログに出ていた `beam=13 max-active=7000 lattice-beam=4`）。**大きいほど精度寄り・遅い**、小さいほど速い。まさに**速度と精度のトレードオフ**をここで調整する。

### ③ 語彙を絞る（グラマー／限定語彙）
- 認識する語彙を**あらかじめ限定**できる（`vosk_recognizer_new_grm` 系）。「数字だけ」「決まったコマンドだけ」など用途が限定的なら、**誤認識が激減**する。メモのような自由発話には不向きだが、限定UIには有効。

### ④ 後処理に使える出力
- **単語ごとの信頼度**（`setWords`）や **n-best（複数候補）**（`SetMaxAlternatives`）を出せる。信頼度の低い語を弾く・候補から選ぶ等の後処理に使える。

### ⑤ 言語モデル適応・rescoring（上級）
- 独自語彙・専門用語に合わせて**言語モデルを適応**したり、軽いモデルで素早く出して**大きいモデルで採点し直す（rescoring）**手法もある（公式の lm / adaptation ドキュメント）。
- 公式の「精度デバッグ」ページは、まず**原因の切り分け（音質・語彙・モデル適合）→ モデル適応**を勧めている。

> momeo 的まとめ：手軽に効くのは **①モデルサイズ** と **②デコードパラメータ**。自由発話のメモ用途では ③語彙限定 は使いにくい。ただし**根本的な日本語精度は Kaldi 系の限界**もあり、sherpa(ReazonSpeech) の方が上だった（実測どおり）。

## 4.【質問3】対応言語

- **40言語以上**。各言語に小モデル（モバイル向け）と大モデル（サーバ向け）。
- 例（小モデル目安）：英40MB / 中42MB / 仏41MB / 独45MB / 西39MB / 露45MB / 韓82MB / 日48MB ほか、伊・葡・蘭・アラビア・トルコ・越・ヒンディー・ポーランド・ウクライナ 等。
- 実装的には **モデルのURL（or パス）を差し替えるだけ**で言語を切替できる（最も手軽な多言語対応）。
- 補足：日本語で見える「分かち書き（単語が空白区切り）」は**日本語特有**。英・仏など元々スペース区切りの言語では自然な出力で、学習資源が多い分 精度も出やすい。

## 5.【質問4】Flutter での使い方

### パッケージ（permission_handler のバージョンに注意）
| パッケージ | permission_handler 依存 | 備考 |
|---|---|---|
| `vosk_flutter` | ^10 | 古い。pmh^12 のプロジェクトとは**衝突** |
| `vosk_flutter_2` | ^11 | これも pmh^12 と**衝突** |
| **`vosk_flutter_service`** | ^12 | **momeo はこれを採用**（pmh^12 互換） |

> momeo は permission_handler ^12 を使うため、`vosk_flutter`/`_2` は version solving で弾かれ、`vosk_flutter_service` のみ導入できた。

### 基本の流れ（SpeechService：マイクお任せ・最も簡単）
```dart
final vosk = VoskFlutterPlugin.instance();

// モデルを自動DL＆解凍（URLを渡すだけ。キャッシュされ2回目以降は即返る）
final modelPath = await ModelLoader()
    .loadFromNetwork('https://alphacephei.com/vosk/models/vosk-model-small-ja-0.22.zip');

final model = await vosk.createModel(modelPath);
final recognizer = await vosk.createRecognizer(model: model, sampleRate: 16000);
final speech = await vosk.initSpeechService(recognizer);

speech.onPartial().listen((json) { /* {"partial":"..."} 暫定 */ });
speech.onResult().listen((json)  { /* {"text":"..."} 確定 */ });

await speech.start();   // 認識開始（マイクは Vosk が掴む）
// await speech.stop();  // 停止
```
- **モデル配置がラク**：`loadFromNetwork` が DL＆解凍＆キャッシュまで面倒を見る（sherpa のような adb 手動配置が不要）。
- 出力は **JSON文字列**（`{"partial":"..."}` / `{"text":"..."}`）。`jsonDecode` して取り出す。

### もう一つの流れ（Recognizer 手動給餌：pre-roll 等のカスタム向け）
- `SpeechService` を使わず、**自分で録音した PCM** を `recognizer.acceptWaveform(bytes)` に流す方式。
- 「最初の言葉が切れる問題」の **先読みバッファ**や、VAD との組み合わせをやりたいときはこちら。

## 6.【質問5】最近の評判・このエンジンは成長しているか

### 開発の活発さ（GitHub: alphacep/vosk-api, 2026-06時点）
- ⭐ 約 **14,800 stars** / fork 約1,700 / open issues 約590。**アーカイブされておらず**、直近（2026-06）にもコミットあり＝**メンテは継続中**。
- ただし**正式リリースは年単位の間隔**（最新 v0.3.50＝2024-04、その前 v0.3.45＝2022-12）。＝**「成熟・安定」フェーズ**で、急成長というより“枯れて使える定番”。

### 技術的な立ち位置と評判
- ベースの **Kaldi は“前世代”の技術**。音声認識の最前線は **end-to-end（Transducer / Attention / LLM結合）** に移っており、**精度では Whisper・sherpa(Zipformer)・NVIDIA 系に劣る**のが一般的評価。
- 一方で **「軽量・オフライン・多言語・真のストリーミング・CPUのみ・組み込み可」** という**運用面の強み**は健在で、「制約の厳しい端末／組み込み／多言語を手早く」用途では**今でも定番**。
- よく言われる弱点：**固有名詞・メールアドレス・数字など“崩れると困る情報”の誤りが多い**。自由会話の大意は取れるが、正確なテキストが要る用途では不利。

> 一言でいうと「**伸び盛りではないが、枯れて信頼でき、軽さと多言語で今も選ばれる**」エンジン。最先端の精度を狙う流れとは別軸。

## 7.【質問6】似たオンデバイス・ストリーミングengineと比較

「端末内で・話しながら出る」系の主な選択肢：

| エンジン | 方式 | 精度 | 速度/遅延 | 多言語 | サイズ | ライセンス/備考 |
|---|---|---|---|---|---|---|
| **Vosk** | Kaldi 系・ストリーミング | 中〜低 | 低遅延・軽い | **40+言語** | 小（〜50MB） | 無料・組み込み定番。固有名詞/数字に弱い |
| **sherpa-onnx（Zipformer transducer）** | next-gen Kaldi・ストリーミング | **高** | 低遅延・速い | 多（ただし**日本語のstreamingモデルは無し**） | 中 | 無料・話者分離など多機能。日本語は offline(ReazonSpeech) が高精度 |
| **Picovoice Cheetah/Leopard** | 商用・ストリーミング | 中〜高 | 低遅延 | 限定的 | 小 | **商用ライセンス必須**（無料枠あり） |
| **OS標準**（Android SpeechRecognizer / iOS Speech） | OS依存 | 中〜高 | 低遅延 | OS依存 | 0（同梱） | 無料だが**自動停止などの制約**（momeo が最初に断念した方式） |
| **whisper.cpp ストリーミング** | Whisper を擬似ストリーム化 | 高 | **遅い** | 99言語 | 中〜大 | 真のストリーミングではない・重い |

### Vosk はどうか（比較の結論）
- **「真のストリーミング × 多言語 × 軽量 × 無料 × 組み込み」**の総合では、**Vosk は今も独自ポジション**（特に多言語を手早く・小サイズで）。
- ただし **純粋な精度**では **sherpa-onnx（Zipformer）に明確に劣る**。日本語に限れば、momeo の実測でも sherpa(ReazonSpeech) の方が**速く・正確・自然**だった（Vosk は分かち書き＋誤認識）。
- **「話しながら出るライブ感」**は Vosk（と sherpa の online モデル）の魅力だが、**日本語ではその online モデルが sherpa に無い**ため、「日本語で真のストリーミングを今すぐ」なら **Vosk がほぼ唯一の現実解**。精度を取るなら sherpa の「即確定（~50ms）」。

## 8. momeo での位置づけ（まとめ）

- Vosk の価値＝**日本語で“話しながら逐次表示”できる数少ない手段**。体験としてのライブ感は唯一。
- ただし**メモ用途で重要な「綺麗で正確な確定テキスト」では sherpa(ReazonSpeech) が上**。momeo の本命候補は sherpa、Vosk は「ライブ表示が欲しい場合の対抗馬」。
- もし Vosk を使うなら、本ガイドの対策が効く：
  - 頭欠け → **先読みバッファ＋自前給餌**（or 一拍おく／常時リスニング）
  - 精度 → **モデルサイズ↑** と **デコードパラメータ**（速度とトレードオフ）
  - 多言語 → **モデルURL差し替え**
