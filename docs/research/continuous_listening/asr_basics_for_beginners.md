## 0. 3行まとめ

- **音声認識エンジン**とは「**モデル（賢さの本体）＋ それを動かす実行プログラム（ランタイム）**」のセット。whisper / sherpa / Vosk はこの“動かす側”の名前で、中で使うモデルは差し替えられる。
- **世界の主流**は2つの世界線：**クラウドAPI**（Google/Deepgram/OpenAI Whisper など・高精度で手軽、要ネット・課金・プライバシー懸念）と、**オンデバイス**（Whisper/NVIDIA/sherpa/Vosk など・オフラインで無料・端末性能の制約あり）。**最高精度は今クラウド寄りの大型モデル**（例: NVIDIA Canary-Qwen）だが、**スマホで動かすなら小型のオンデバイス**を選ぶ。
- **Flutter への組み込み**は「**パッケージ（＝橋渡し）＋ モデルファイル（別途DL/同梱）＋ マイク取得＋VAD**」の組み合わせ。パッケージを入れるだけで完結しないことが多い（モデルの配置が要る）。

## 1. そもそも「エンジン」とは何か

身近な例え：**料理**で考える。

| 料理の世界 | 音声認識の世界 | 役割 |
|---|---|---|
| レシピ＆料理人の腕（知識） | **モデル** | 「この音はこの文字」と判断する“賢さの本体”。学習で作られたデータの塊（数十MB〜数GB） |
| キッチン・調理器具 | **ランタイム（推論エンジン）** | モデルを実際に動かす実行プログラム。CPU/GPU を使って計算する |
| 「材料を入れたら料理が出てくる」仕組み全体 | **エンジン** | 音声を入れたらテキストが出てくる、モデル＋ランタイム＋前後処理のまとまり |

つまり：

- **モデル ＝ 賢さの本体**（ファイル。例: `ggml-small.bin`、`encoder.onnx`）。
- **ランタイム ＝ それを動かす土台**（例: `whisper.cpp`、`onnxruntime`、`Kaldi`）。
- **エンジン ＝ その両方をまとめた“音声→文字”の装置**。

今回の3つを当てはめると：

| 呼び名 | 正体（ランタイム） | 中で使うモデル | 特徴 |
|---|---|---|---|
| **whisper**（whisper_flutter_new） | `whisper.cpp` | OpenAI Whisper の ggml モデル | 高精度・バッチ向き・やや重い |
| **sherpa(-onnx)** | `onnxruntime` ＋ next-gen Kaldi | Zipformer / SenseVoice 等の onnx モデル | 高速・モデル差し替えが豊富 |
| **Vosk** | （旧来の）Kaldi | Vosk 各言語モデル | 真のストリーミング・軽量・精度は控えめ |

ポイント：**「エンジン」と「モデル」は別物**。同じ sherpa（ランタイム）でも、載せるモデルを変えれば日本語特化にも多言語にもなる（前の文書の SenseVoice/Canary の話はこれ）。

## 2. エンジンの“種類”（中身のアーキテクチャ）

モデルの作り（アーキテクチャ）によって、得意・不得意が変わる。難しい名前が出てくるが、**「ざっくりこういう性格」**だけ掴めば十分。

### 2-1. 認識方式（4タイプ）

| 方式 | ざっくり性格 | 代表 | ストリーミング |
|---|---|---|---|
| **CTC** | シンプル・速い。文脈の理解は弱め | wav2vec2, Parakeet CTC | しやすい |
| **Transducer（RNN-T / TDT）** | **話しながら逐次出せる**。速度と精度のバランス良 | Zipformer（sherpa）, Parakeet TDT, Kaldi/Vosk | **得意（これが“ライブ字幕”の主役）** |
| **Attention（encoder-decoder）** | **精度が非常に高い**が、まとまった区間を一気に処理＝**遅延が出る** | **Whisper**, Canary | 苦手（Whisper は30秒単位） |
| **LLM結合（SALM）** | 音声エンコーダ＋大規模言語モデル。**今いちばん高精度**だが重い | NVIDIA Canary-Qwen | 苦手・重い |

> 今回 whisper が「話し終えてから数秒」だったのは Attention 方式の宿命。sherpa（Transducer）が速かったのも、Vosk が“話しながら出た”のも Transducer/Kaldi 系だから。**方式の違いがそのまま体験の違いになった**わけ。

### 2-2. 2つの大きな軸

**① ストリーミング（オンライン） vs バッチ（オフライン）**

- **ストリーミング**＝喋っている最中に文字が出る（低遅延）。例: Vosk、Transducer 系。
- **バッチ**＝ある区間をまとめて処理して確定（高精度だが遅延あり）。例: Whisper。

**② オンデバイス vs クラウド**

| | オンデバイス（端末内） | クラウド（サーバー） |
|---|---|---|
| 精度 | 端末性能の制約で中〜高 | **最高クラス**（大型モデルが使える） |
| ネット | **不要（オフライン）** | 必要 |
| お金 | **無料**（端末で計算） | 従量課金が多い |
| プライバシー | **音声が外に出ない** | 音声を外部送信 |
| 手軽さ | モデル配置など手間あり | API を叩くだけで楽 |

> momeo は「プライバシー重視・常時録音・個人開発（低コスト）」なので **オンデバイス**を選んでいる。その制約の中で速度・精度を探したのが一連の検証。

## 3. 世界では何が使われ、今の最高性能は何か（2026年時点）

### 3-1. クラウドAPI（手軽・高精度・要ネット/課金）

| 提供 | 特徴 |
|---|---|
| **OpenAI Whisper API** | 定番。多言語・高精度。安価（約 $0.006/分） |
| **Deepgram（Nova-3 / Flux）** | リアルタイム特化（<300ms）。音声エージェント向け |
| **AssemblyAI（Universal-2）** | 高精度（約 8.4% WER）＋要約など“理解”機能 |
| **ElevenLabs Scribe v2 Realtime** | 30言語・150ms の低遅延リアルタイム |
| **Google / Azure / AWS Transcribe** | 大手クラウドの定番。SLA・周辺サービスが強い |

### 3-2. オープン／自前運用・オンデバイス系

| 名前 | 何者か |
|---|---|
| **Whisper（OpenAI）** | オープンの“事実上の標準”。多言語に強い。派生多数（whisper.cpp, faster-whisper） |
| **NVIDIA NeMo（Parakeet / Canary）** | 近年トップ級。**Parakeet=速い**、**Canary=高精度**。Canary-Qwen は LLM 結合で首位 |
| **sherpa-onnx（next-gen Kaldi / Zipformer）** | 軽量・高速・組み込み向き。今回採用候補 |
| **Vosk（Kaldi）** | 老舗。軽量・40言語超・ストリーミング。精度は控えめ |
| **SenseVoice（Alibaba FunASR）** | 中英日韓＋広東語を1モデルで。高速 |

### 3-3. 「今いちばん高精度」は？

英語中心の客観ベンチ **Open ASR Leaderboard（Hugging Face）** では（2026年時点）：

- **1位 NVIDIA Canary-Qwen 2.5B（平均 WER 約 5.6%）** … 音声エンコーダ＋LLM（Qwen）結合の重量級。
- 僅差で **IBM Granite Speech 3.3 8B（約 5.85%）**。
- **速度重視なら NVIDIA Parakeet TDT 系**（精度トップ群よりは落ちるが圧倒的に速い・リアルタイム向き）。

> 教訓：**「最高性能の単一モデル」は存在しない**。「精度トップ（重い）」と「速度トップ（軽い）」はトレードオフ。**用途で選ぶ**もの。momeo は「スマホでオフライン・低遅延・日本語」なので、巨大な Canary-Qwen ではなく、軽量・高速・日本語特化の **sherpa + ReazonSpeech** が刺さった、という位置づけ。

参考: WER（Word Error Rate）＝ 単語誤り率。低いほど高精度。

## 4. Flutter アプリへの組み込み方（「パッケージ入れるだけ」では足りない理由）

### 4-1. なぜ“橋渡し”が要るのか

音声認識エンジンの中身は **C/C++ で書かれた重い計算プログラム**（＋モデルファイル）。一方 Flutter のコードは **Dart** という別の言語で動く。Dart は、こうしたネイティブの重い処理を**直接は実行できない**。

そこで **Flutter プラグイン（パッケージ）が“橋渡し”**をする：

```
[ Dart（あなたのアプリのコード） ]
        │  ← パッケージが用意する橋
        ▼
[ ネイティブのエンジン本体（C/C++ の .so / .framework） ]
```

橋のかけ方は主に2種類（どちらでも結果は同じ。利用者は意識しなくてよい）：

- **FFI**：Dart から C 関数を直接呼ぶ（例: whisper_flutter_new, sherpa_onnx）。
- **プラットフォームチャネル**：Dart ↔ Kotlin/Swift でメッセージをやり取りする（例: Vosk の一部）。

### 4-2. 「パッケージ＝橋＋エンジン本体」だが、“モデル”は別

ここが初心者がつまずく所：

- パッケージを入れると、**橋＋エンジンのネイティブ本体（ランタイム）**は手に入る。
- でも **モデルファイル（賢さの本体・数十MB〜数百MB）は普通パッケージに入っていない**。アプリが大きくなりすぎるから。
- だから実際には **モデルを別途用意**する必要がある：
  - **実行時にダウンロード**（例: whisper は HuggingFace から、Vosk は `loadFromNetwork` で自動DL）、
  - または **アプリに同梱（assets）** したり、開発中は **adb で端末に配置**したり。

> 今回 sherpa が「パッケージは入ったのにモデルが無くて動かない」状態になり、adb でモデルを置いたのはこのため。Vosk は `loadFromNetwork` が自動DLしてくれるので楽だった。

### 4-3. クラウドAPIの場合は話が別

クラウドAPIを使う場合は、端末にエンジンもモデルも要らない。**パッケージ（or 単なる HTTP 通信）で音声をサーバーに送り、結果のテキストを受け取る**だけ。手軽だがネット必須・課金・音声を外部送信、という前述のトレードオフになる。

## 5. 音声を取る部分と「VAD」とは

エンジンは「音声データ」を受け取って「テキスト」を返すが、**その“音声データ”を用意するのは別の仕事**。

### 5-1. マイク → PCM

- マイクで拾った音は、最終的に **PCM** という“生の数値の波形データ”になる（音の大きさを細かい時間間隔で数値化したもの）。
- 音声認識では **16kHz・モノラル・16bit** が定番（1秒を16000個の数値で表す）。
- この PCM を作る担当：`record` のような**録音パッケージ**、あるいはエンジン付属のマイク機能（Vosk の `SpeechService`）、あるいは **`vad` パッケージ**（マイクを掴んで PCM を流しつつ発話検出もする）。

### 5-2. VAD（Voice Activity Detection ＝ 発話区間検出）とは

- **「今しゃべっている／無音」を判定する仕組み**。小さな機械学習モデル（Silero VAD など）で、短いフレームごとに「これは音声っぽい」確率を出す。
- 何の役に立つか：
  1. **区切り**：連続録音を「ひと続きの発話」に切り分ける（無音が続いたら“発話終了”）。
  2. **トリガー**：「発話終了 → ここでエンジンに転写させる」という合図になる。
  3. **節約**：無音のときはエンジンを動かさず、電池・計算を節約。
- ※ VAD は「**しゃべっているかどうか**」を見るだけで、**中身（言葉）は分からない**。言葉に変えるのはエンジンの仕事。

### 5-3. 全体の流れ（パイプライン）

```
🎤 マイク
   │  音をPCM（生の数値波形）に
   ▼
[ 録音 / VAD ]  ← VADが「発話の終わり」を見つける
   │  ひと続きの発話（音声チャンク）
   ▼
[ 音声認識エンジン ]  ← whisper / sherpa / Vosk
   │  テキスト
   ▼
💾 保存（メモとして）
```

### 5-4. momeo の各構成での役割分担

| 部品 | 役割 |
|---|---|
| `vad`（Silero VAD） | マイク所有＋PCM供給＋発話区切り（whisper/sherpa 構成で使用） |
| `whisper_flutter_new` | whisper エンジン（バッチ転写） |
| `sherpa_onnx` | sherpa エンジン（高速・日本語特化モデル） |
| `vosk_flutter_service` | Vosk エンジン（**自分でマイクも持つ**ので VAD 不要・真のストリーミング） |

> Vosk だけ「マイク取得＋区切り＋認識」を1パッケージで完結する点が他と違う。whisper/sherpa は「マイク＆区切り（vad）」と「認識（エンジン）」を分担している。

## 6. 用語集（ミニ辞書）

| 用語 | やさしい意味 |
|---|---|
| **STT / ASR** | Speech To Text / Automatic Speech Recognition。どちらも「音声→文字」 |
| **モデル** | 学習で作った“賢さの本体”。ファイルの塊 |
| **ランタイム / 推論エンジン** | モデルを動かす実行プログラム（whisper.cpp, onnxruntime, Kaldi） |
| **PCM** | 生の音声データ（数値の波形）。16kHz/モノラルが定番 |
| **VAD** | 発話か無音かを判定する仕組み（言葉の中身は見ない） |
| **ストリーミング/オンライン** | 喋りながら逐次テキストが出る方式 |
| **バッチ/オフライン処理** | 区間をまとめて処理して確定する方式（≠ オフライン＝ネット不要、と紛らわしいので注意） |
| **オンデバイス** | 端末内で完結（ネット不要・無料・プライバシー◎） |
| **クラウドAPI** | サーバーに音声を送って結果を受け取る方式 |
| **CTC / Transducer / Attention** | 認識方式の種類。Transducer はストリーミング向き、Attention は高精度だが遅延 |
| **WER** | Word Error Rate（単語誤り率）。低いほど高精度 |
| **RTF** | Real Time Factor。1秒の音声を処理するのにかかる秒数。0.05 なら 20倍速 |
| **FFI** | Dart から C 関数を直接呼ぶ橋渡しの仕組み |
| **プラットフォームチャネル** | Dart ↔ Kotlin/Swift のメッセージ橋渡し |
| **ONNX** | 機械学習モデルの共通フォーマット（onnxruntime で動かす） |
| **ggml / gguf** | whisper.cpp 系で使うモデル形式 |
| **量子化（int8 等）** | モデルを軽く・速くする圧縮。精度は少し落ちることがある |

## 7. momeo での位置づけ（まとめ）

- momeo は **オンデバイス・日本語・低遅延** が要件。だから巨大な最高精度モデル（クラウドの Canary-Qwen 等）ではなく、**スマホで動く軽量エンジン**を比較してきた。
- 方式の違い（Attention=遅い / Transducer=速い）が、そのまま体験差になった：
  - whisper（Attention）＝高精度だが「話し終えて数秒」。
  - **sherpa（Transducer・日本語特化）＝ ~50msで即確定。現状ベスト**。
  - Vosk（Kaldi・ストリーミング）＝話しながら出るが精度・分かち書きが弱い。
- 将来の多言語化も、**エンジンはそのままモデルを差し替える**形で対応できる（sherpa の SenseVoice/Canary、Vosk の言語別モデル）。詳細は `vad_whisper_impl_log.md` §8。
