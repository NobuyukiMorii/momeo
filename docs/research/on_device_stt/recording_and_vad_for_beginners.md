# 「sherpa 内蔵 VAD + record」って何？ — 素人開発者向けやさしい解説

## この文書について

- **目的**: 実装計画 `docs/on_device_stt/outline.md` に出てくる **「`sherpa_onnx` 内蔵の Silero VAD ＋ `record`」** という構成が、何を指していて、なぜそれを選んだのかを、専門外でも分かるように噛み砕いて説明する。
- **対象読者**: この構成がピンと来ていない開発者（＝この文書を依頼した本人）。
- **作成日**: 2026-06-20
- **前に読むとよい**: `asr_basics_for_beginners.md`（エンジン・モデル・VAD の超基礎）。本書はその続きで、「**録音とVADの“担当者”をどう組むか**」に絞って説明する。
- 用語は本文で都度説明し、最後に**用語集（追加分）**もある。

---

## 0. 3行まとめ

- 音声をテキストにするには「**①マイクで録音 → ②しゃべってる区間を切り出す（VAD）→ ③その区間を文字にする（ASR）**」の3つの仕事がいる。
- 「**sherpa 内蔵 VAD + record**」とは、**①録音を `record` という部品に、②区切り（VAD）と③文字化（ASR）を `sherpa_onnx` という1つの部品に**まかせる、という“担当の割り振り”のこと。
- 以前は②の区切りを `vad` という別部品にやらせる計画だったが、それだと**同じ部品（onnxruntime）が二重に入ってビルドが壊れる**などの問題があった。**②も sherpa にまかせる**ことで、その問題が丸ごと消えた。これが案Cの正体。

---

## 1. まず「3つの仕事」を思い出す

音声メモを作るには、ざっくり次の3ステップが必要（詳細は `asr_basics_for_beginners.md` 参照）。

```text
① 録音       … マイクから「生の音の数値（PCM）」を取り続ける
② 区切り(VAD) … その音から「今しゃべってる／無音」を判定し、ひと続きの発話を切り出す
③ 文字化(ASR) … 切り出した発話を「日本語テキスト」に変換する
```

- **①は耳**（音を拾う）
- **②は「発言の始まりと終わりを見つける受付係」**（言葉の中身は分からない。しゃべってるかどうかだけ見る）
- **③は「聞いた言葉を書き取る書記」**（中身を文字にする）

この3つを **どの部品（パッケージ）に担当させるか** が「構成」の話。今回はそこを変えた。

---

## 2. 登場人物（部品）を一言ずつ

| 名前 | 種類 | 一言でいうと | 担当する仕事 |
|---|---|---|---|
| **`record`** | Flutter パッケージ | マイクから音を取り続ける録音係 | ①録音 |
| **Silero VAD** | 小さなAIモデル（約2MB の `silero_vad.onnx`） | 「しゃべってる／無音」を見分ける受付係の“頭脳” | ②区切り |
| **`sherpa_onnx`** | Flutter パッケージ（エンジン） | AIモデルを動かす“何でも屋”。VAD も ASR も動かせる | ②区切り＋③文字化 |
| **NeMo parakeet CTC** | 大きなAIモデル（約625MB の `model.int8.onnx`） | 日本語を書き取る書記の“頭脳” | ③文字化 |
| **onnxruntime** | 計算ライブラリ（裏方） | 上のAIモデルたちを実際に計算する“エンジンルーム” | ②③の足回り |

ポイント：
- **`sherpa_onnx` は「箱（エンジン）」**で、その中に **VAD用のモデル（Silero）** と **文字化用のモデル（NeMo）** の両方を入れて動かせる。
- **onnxruntime は普段は意識しない裏方**だが、後で「これが二重に入ると困る」という主役級の話になる（§5）。

---

## 3. 「sherpa 内蔵 VAD + record」を一文で

> **録音は `record` にやらせ、②区切りと③文字化はどちらも `sherpa_onnx` の中で（内蔵の Silero VAD と NeMo モデルで）やる構成。**

「内蔵 VAD」＝ `sherpa_onnx` が**最初から VAD の機能を持っている**こと。だから区切り専用の別パッケージを足さなくていい。

---

## 4. たとえ話：会議の議事録チーム

「会議をリアルタイムで議事録にするチーム」で考えると分かりやすい。

| チームの役 | 部品 | やること |
|---|---|---|
| **マイク係** | `record` | 会議の音をずっと録り続けて、書記チームに渡す |
| **受付係** | sherpa 内蔵の Silero VAD | 「今だれか話し始めた／話し終わった」を見張り、“ひと続きの発言”の区切りで書記に「はい、ここまで！」と渡す |
| **書記** | sherpa + NeMo モデル | 渡された発言を日本語の文章に書き取る |

- マイク係（`record`）は**ずっと録り続ける**だけ。区切りは判断しない。
- 受付係（VAD）は**言葉の意味は分からない**。「声がしてる／してない」だけを見て、発言の切れ目を作る。
- 書記（ASR）は**切り出された発言だけ**を文字にする。

**そして案Cでは、受付係と書記が“同じ会社（sherpa）の社員”**。以前の計画では受付係だけ別会社（`vad` パッケージ）から呼んでいた。これが後で問題になった（§5）。

---

## 5. なぜ変えたのか（旧計画 → 案C）

### 5-1. 旧計画：区切りを `vad` という別部品にまかせていた

```text
record/vad ── ① 録音
   vad     ── ② 区切り（vad パッケージが Silero VAD を動かす）
 sherpa    ── ③ 文字化（NeMo モデル）
```

一見よさそうだが、**`vad` パッケージも `sherpa_onnx` も、裏で同じ「onnxruntime」という計算ライブラリを自前で抱えている**。両方アプリに入れると、onnxruntime が**2つ**入ってしまう。

### 5-2. 起きる問題：「libonnxruntime.so の二重同梱」

`libonnxruntime.so` とは、onnxruntime の本体ファイル（Android 用の部品）。これが**2つ入る**と、Android のビルドが **「同じ名前の部品が2つある。どっちを使えばいいか分からない」** とエラーで止まる。

> たとえ：引っ越しで、同じ「冷蔵庫」を2台運び込もうとして、搬入係が「2台は置けません！」と止まる感じ。

これを回避する“応急処置”（どちらか1台だけ使う、と手動指定する `pickFirst` / `exclude` 設定）はあるが、

- バージョン違いの冷蔵庫が混ざって**起動時に壊れる**危険、
- そもそも**応急処置が必要なこと自体が不格好**、

という難点があった。旧 `outline.md` が「乱暴」と書いていたのはこれ。

### 5-3. 案C：区切りも sherpa にまかせれば、そもそも問題が起きない

```text
record  ── ① 録音
 sherpa ── ② 区切り（内蔵 Silero VAD）
 sherpa ── ③ 文字化（NeMo モデル）
```

`vad` パッケージを使わないので、**onnxruntime は sherpa の中の1つだけ**になる。

> 冷蔵庫が最初から1台しかないので、搬入係はそもそも困らない。応急処置（`pickFirst` 等）も要らない。

これが「案Cが本質的にクリーン」と言っている意味。実際に検証ブランチで試したら、応急処置を一切入れずに Android ビルドが通った（`docs/on_device_stt/verification/README.md`）。

### 5-4. ついでに良くなったこと（おまけの利点）

| 観点 | 旧計画（vad あり） | 案C（vad なし） |
|---|---|---|
| onnxruntime の数 | 2つ（要・応急処置） | **1つ（応急処置不要）** |
| iOS の対応バージョン | iOS **15.1 以上**に引き上げが必要だった | **iOS 13.0 のままでOK**（古い端末も対応維持） |
| VAD モデルの入手 | `vad` は初回にネットからDLする（`INTERNET` 権限が要る） | **`silero_vad.onnx` をアプリに同梱**（ネット不要・完全オフライン） |
| 必要なマイク権限 | RECORD_AUDIO ＋ 余分な権限 | **RECORD_AUDIO だけ** |

- **iOS 15.1 の話**：`vad` が中で使う別部品が「iOS 15.1 以上じゃないと動かない」と要求していた。`vad` を外したのでこの縛りが消え、**iOS 13/14 の古い端末も切り捨てずに済む**。
- **オフラインの話**：`vad` は区切り用モデルを初回起動時に**インターネットから落とす**仕組みだった。案Cは小さな `silero_vad.onnx`（約2MB）を**最初からアプリに同梱**するので、ネットが無くても初回から動く。

## 6. 全体の流れ（案Cのパイプライン）

```text
🎤 マイク
   │
   ▼
[ record ]  ← ① ずっと録音（PCM16 / 16kHz / モノラル の生データ）
   │  音の数値をかたまりで渡す
   ▼
[ sherpa 内蔵 Silero VAD ]  ← ② 「発話の始まり〜終わり」を見つけて1発話を切り出す
   │  ひと続きの発話（音声チャンク）
   ▼
[ sherpa + NeMo モデル ]  ← ③ その発話を日本語テキストに書き取る
   │  確定テキスト
   ▼
💾 メモカードとして保存・表示（_addMemo に渡す）
```

- `record` が出すのは「PCM」という**生の音の数値**（`asr_basics_for_beginners.md` §5 参照）。
- それを sherpa の VAD に渡すと、しゃべり終わったタイミングで**1発話分**をまとめてくれる。
- その1発話を sherpa の書記（NeMo）に渡すと、**日本語の文章**が返る。実機では1発話あたり**約75〜215ミリ秒**で返ってきた（ほぼ一瞬）。

## 7. `outline.md` を読むための“翻訳表”

`outline.md` に出てくる表現と、本書のやさしい言い換えの対応。

| outline.md の表現 | やさしい意味 |
|---|---|
| sherpa 内蔵の Silero VAD | sherpa が最初から持っている「発話の区切り」機能 |
| `record` で PCM ストリームを取得 | `record` でマイクから生の音データを取り続ける |
| PCM16 / 16kHz / モノラル | 音声認識で定番の音データ形式（1秒を16000個の数値で、1チャンネルで表す） |
| Float32 に変換して窓ごとに供給 | 生データを VAD が食べやすい形・大きさに整えて渡す（細かい技術都合） |
| `SpeechSegment`（発話チャンク） | VAD が切り出した「ひと続きの発話」のかたまり |
| `OfflineRecognizer(nemoCtc)` | NeMo モデルで文字化する書記の設定 |
| onnxruntime が1本になる | 裏方の計算ライブラリが1つだけになり、ビルドが安定する |
| pickFirst / exclude が不要 | 「冷蔵庫2台問題」の応急処置がそもそも要らない |
| デプロイメントターゲット 13.0 据え置き | iOS の対応最低バージョンを上げなくてよい（古い端末も対応） |

## 8. ミニ用語集（本書で増えた分）

| 用語 | やさしい意味 |
|---|---|
| **`record`** | マイクから録音する Flutter パッケージ |
| **内蔵 VAD** | エンジン（sherpa）が最初から持っている発話区切り機能。別パッケージを足さなくていい |
| **`silero_vad.onnx`** | 区切り用の小さなAIモデル（約2MB）。これをアプリに同梱する |
| **onnxruntime / `libonnxruntime.so`** | AIモデルを計算する裏方ライブラリ（とその Android 用本体ファイル） |
| **二重同梱** | 同じ部品がアプリに2つ入ってしまうこと。ビルドが止まる原因 |
| **pickFirst / exclude** | 二重同梱の応急処置（手動でどちらか1つに絞る設定）。案Cでは不要 |
| **デプロイメントターゲット** | アプリが対応する OS の最低バージョン（iOS 13.0 など） |
| **PCM** | 生の音データ（数値の波形）。16kHz/モノラルが定番 |
| **発話チャンク / SpeechSegment** | VAD が切り出した「ひと続きの発話」のかたまり |

---

## 9. まとめ（これだけ覚えれば十分）

- やることは **①録音 → ②区切り → ③文字化** の3つ。
- 案C ＝ **①は `record`、②と③は `sherpa_onnx`（内蔵VAD＋NeMoモデル）にまかせる**。
- 区切り専用の `vad` パッケージを使わないことで、**裏方（onnxruntime）が1つで済み、ビルドが安定・iOSも古い端末対応・完全オフライン・権限も最小**になった。
- これらは**実機で実証済み**（`docs/on_device_stt/verification/README.md`）。だから `outline.md` はこの案Cを前提に書かれている。
