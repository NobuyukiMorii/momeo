Flutterで音声メモアプリのようなものを作っていて、「マイクの音を録音する」ところまではできました。

でも、その次で手が止まりました。

録音して取れるのは、ずっとつながった**一本の音**です。その中には、しゃべった部分も、黙っている部分も、考え込んでいる間も、全部まとめて入っています。

僕がやりたかったのは、この一本の音を **「1回しゃべったぶん」ごとに区切る** ことでした。

```text
録音した音:  [ 無音…「今日はいい天気」…無音…「買い物に行こう」…無音 ]

やりたいこと: 「今日はいい天気」 と 「買い物に行こう」 を別々に取り出す
```

最初は「録音できたんだから、無音のところで切ればいいだけでしょ？」と軽く考えていました。でも、いざやろうとすると「どこからが声で、どこからが無音なのか」を自分で判定するのが意外と難しい。エアコンの音や物音もあるし、息継ぎの一瞬の静けさで切れてしまっても困ります。

そこで出てくるのが **VAD** です。この記事では、VADという言葉を初めて聞く人にも分かるように、「録音した音を発話ごとに区切る」のに何をしたのかを整理してみます。

使ったのは、前に調べた `sherpa-onnx`。実は音声認識だけでなく、この「区切り」もこのライブラリの中だけで完結できました。

## 30秒でいうと

- ずっと録音した音は、声・無音・物音が一本につながっている。これを **「1回の発話」ごとに区切りたい**。
- 区切ってくれるのが **VAD（声がある／ないを判定する仕組み）**。今回は **Silero VAD** を使った。
- VADは `sherpa-onnx` に**最初から入っている**ので、別のライブラリは要らなかった。
- 必要なモデルは `silero_vad.onnx` という **約0.6MBの小さなファイル1つだけ**。音声認識用の重いモデル（数百MB）はこの段階では不要。
- マイクの音を流し込むと、**「ひと続きの発話のかたまり」が1個ずつ出てくる**。区切り方は設定値（何秒 黙ったら終わりとみなすか、など）で調整できる。

では順番に。

## やりたいこと：一本の音を「発話ごと」に切り分ける

音声メモアプリ全体で見ると、処理はこういう流れになります。

```text
マイク → 録音 → 区切り → 文字にする → メモとして保存
              ↑
              この記事はここだけ
```

「録音」までは `record` というパッケージでできました（マイクの音を生データとして取り出す部分）。「文字にする」は音声認識モデルの仕事で、これはまた別の段階です。

この記事で扱うのは、その真ん中の **「区切り」** です。

なぜわざわざ区切るのか。理由はシンプルで、**ずっと録音した音をまるごと文字化に渡すより、発話ごとに切ってから渡した方が扱いやすいから**です。無音や間を一生懸命に文字化しても意味がないですし、「1回しゃべり終わった」というタイミングが分かれば、「よし、ここで文字にしよう」という合図にもなります。

## それをやるのが VAD

**VAD（Voice Activity Detection）** は、ひとことで言うと **「いま声がある？ ない？」だけを判定する仕組み** です。

ポイントは、VADは**中身（何をしゃべったか）までは分からない**こと。あくまで「ここは声」「ここは無音」を見分ける*門番*のような役割です。文字にするのは、その先の音声認識モデルの仕事です。

VADがあると、こういうことができます。

```text
入力（一本の音）:  無音 → 発話A → 無音 → 発話B → 無音

VADを通すと:       発話A
                  発話B          ← 声の部分だけ、ひとかたまりずつ取り出せる
```

「どこからどこまでが1回の発話か」を教えてくれるので、その**かたまり単位**で次の処理に渡せるわけです。

## 区切り方が、そのまま「メモの単位」になる

ここが、触ってみて一番「なるほど」と思ったところです。

音声メモアプリでは、**この区切り1つが、そのままメモ1枚の単位になります**。

```text
発話A → メモカード1枚目
発話B → メモカード2枚目
```

つまり、**区切り方が雑だと、メモの分かれ方も雑になる**ということです。たとえば短く切りすぎると、ひと続きで話したつもりの内容が、細切れのメモにバラバラに割れてしまいます。

```text
「今日はいい天気だから……買い物に行こう」

区切りが短すぎると:
  メモ1:「今日はいい天気だから」
  メモ2:「買い物に行こう」     ← 1つの話のつもりが2枚に割れる
```

なので「どのくらいの間（無音）で区切るか」のさじ加減が地味に重要で、ここは後半の「設定」の話につながります。

## sherpa では VAD が「最初から入っている」

最初、僕は「録音は `record`、VADはVAD専用のライブラリ、音声認識はまた別……」と、機能ごとに別々のライブラリを組み合わせるものだと思っていました。

でも調べてみると、`sherpa-onnx` は音声まわりのいろいろな機能を扱える“道具箱”で、**VADもその中に最初から入っていました**。

なので、やることは「VADモデルを指定して、音を流し込む」だけ。新しいライブラリを足す必要はありませんでした。

Flutterからは、こんなオブジェクトを作って使います（細かい設定は後述）。

```dart
import 'package:sherpa_onnx/sherpa_onnx.dart' as sherpa;

final vad = sherpa.VoiceActivityDetector(
  config: sherpa.VadModelConfig(
    sileroVad: sherpa.SileroVadModelConfig(
      model: modelPath, // VADモデルのファイルパス
    ),
    sampleRate: 16000,
  ),
  bufferSizeInSeconds: 60,
);
```

## 必要なモデルは silero_vad.onnx だけ（約0.6MB）

`sherpa-onnx` は「土台（ランタイム）」で、実際に何をするかは**どのモデルを入れるか**で決まります。VADをやりたいなら、VAD用のモデルが要ります。

今回使ったのは **Silero VAD** という定番のモデルです。ファイル名は `silero_vad.onnx`。

ありがたいことに、これは **約0.6MBしかありません**。音声認識用のモデルは数百MBになることもあって、その「どう配るか」が大きな課題なのですが、VADモデルはとても小さいので、**アプリに普通に同梱するだけ**で済みました。

モデルは sherpa-onnx 公式の配布物から取得できます。

```bash
curl -fSL -o assets/models/silero_vad.onnx \
  https://github.com/k2-fsa/sherpa-onnx/releases/download/asr-models/silero_vad.onnx
```

これを `pubspec.yaml` のアセットに登録しておきます。

```yaml
flutter:
  assets:
    - assets/models/silero_vad.onnx
```

## 小さなつまずき：モデルは「ファイルのパス」で渡す必要がある

ここで1つハマりました。

`sherpa-onnx` にモデルを渡すとき、**アプリに同梱したアセットを直接は読めません**。「端末上の実ファイルのパス」を渡す必要があります。

アセット（`assets/...`）は、アプリの中にパッケージされた読み取り専用のデータで、普通のファイルパスとしては存在しないんですね。

なので、**初回に一度だけ、アセットを端末の書き込み可能な場所へコピーして、そのコピー先のパスを渡す**という手順を踏みます。

```dart
import 'package:flutter/services.dart' show rootBundle;
import 'package:path_provider/path_provider.dart';

Future<String> ensureModel() async {
  final dir = await getApplicationSupportDirectory();
  final file = File('${dir.path}/silero_vad.onnx');

  // まだコピーしていなければ、アセットから書き出す
  if (!await file.exists()) {
    final data = await rootBundle.load('assets/models/silero_vad.onnx');
    final bytes = data.buffer.asUint8List(data.offsetInBytes, data.lengthInBytes);
    await file.writeAsBytes(bytes, flush: true);
  }

  return file.path; // この実パスを VAD に渡す
}
```

0.6MBと小さいので、この方式で **iOS / Android どちらも同じコード**で動きました。

## 区切りの仕組み（箱）は3ステップ

ここからが本題の「マイクの音を入れたら、発話のかたまりが出てくる箱」の中身です。

やっていることは、大きく3つだけでした。

```text
① 音の数値を、VADが読める形に直す
② 決まった量ずつ、VADに渡す
③ 区切れた発話を、取り出す
```

順番に見ていきます。

### ① 音の数値を、VADが読める形に直す

`record` から流れてくる音（PCM）は、`-32768〜32767` の **整数** です。一方、VADが受け取るのは `-1.0〜1.0` の **小数（Float32）**。

なので、**整数を 32768 で割って小数に直します**。

```dart
// 整数（PCM16）→ 小数（Float32）
final sample = int16Value / 32768;
```

やっていることは「数字の表し方を変えるだけ」で、音の中身は変わりません。`32768` は `2の15乗`で、PCM16の整数が取りうる範囲の基準です。これで割ると、ちょうど `-1.0〜1.0` の範囲に収まります。

> 整数で記録するか小数で扱うかは、ただの“目盛りの違い”だと思えば大丈夫です。

### ② 決まった量ずつ、VADに渡す

Silero VAD は、音を**好きなだけまとめて**渡せるわけではなく、**512サンプル（ごく短い一定量）ずつ**受け取る決まりになっています（16kHzのSilero用の単位）。

なので、流れてくる音を**いったん貯めておいて、512サンプルたまるたびに渡す**、というのを繰り返します。

```dart
// 512サンプルたまるごとに VAD へ渡す
while (buffer.length >= 512) {
  final window = Float32List.fromList(buffer.sublist(0, 512));
  buffer.removeRange(0, 512);
  vad.acceptWaveform(window);
}
```

「512」は自分で決めた値ではなく、**Silero VAD側が要求する固定の単位**です。ここは深く考えず、そういう決まりだと思って合わせれば大丈夫でした。

### ③ 区切れた発話を、取り出す

音を渡し続けていると、VADが内部で「1発話ぶん終わった」と判断したかたまりを溜めていきます。それを順番に取り出します。

```dart
while (!vad.isEmpty()) {
  final segment = vad.front(); // 1発話ぶんのかたまり
  vad.pop();                   // 取り出したら捨てる

  // segment.samples が「1発話ぶんの音」
  // 長さ（秒） = サンプル数 ÷ 16000
  final seconds = segment.samples.length / 16000;
}
```

`isEmpty()` で「取り出せるかたまりがあるか」を確認し、`front()` で見て、`pop()` で取り出す——というキューのような使い方です。

この `segment.samples` が、まさに「1回しゃべったぶんの音」。最終的には、これを音声認識モデルに渡して文字にすることになります（それは次の段階の話）。

### 録音を止めるときの注意

1つ気をつけたのが、**録音を止めた瞬間に、まだVADの中に残っている発話**があること。

最後の発話は「無音が続いたら終わり」の無音がまだ来ていないので、放っておくと取り出せません。そこで停止時に `flush()` を呼んで、**残っているぶんを押し出して**から取り出します。

```dart
await recorder.stop();
vad.flush();   // 末尾に残った発話を押し出す
// このあと、もう一度 ③ の取り出しをやる
```

## 区切り方は「設定」で変わる

ここまでで「区切る箱」はできましたが、**どのくらいで区切るか**は設定値で変わります。VADを作るときに、いくつかパラメータを渡せます。

```dart
sherpa.SileroVadModelConfig(
  model: modelPath,
  minSilenceDuration: 1.5,  // 無音がこれだけ続いたら「発話終了」
  minSpeechDuration: 0.25,  // これより短い音は発話と認めない
  maxSpeechDuration: 30.0,  // 一度も黙らず喋り続けた時の強制区切り
);
```

それぞれの意味と、初期値はこうしました。

| パラメータ | 役割 | 大きくすると | 初期値 |
|---|---|---|---|
| `minSilenceDuration` | **主役**。何秒 黙ったら「発話終了」とみなすか | 区切りが減り、1つの発話が長くなる | **1.5秒** |
| `minSpeechDuration` | これより短い音は発話と認めない | 短い物音を拾わなくなる | 0.25秒 |
| `maxSpeechDuration` | **安全弁**。一度も黙らず喋り続けた時だけ強制的に区切る上限 | めったに効かない | 30秒 |

区切りの**主役は `minSilenceDuration`（無音の長さ）** です。「1.5秒くらい黙ったら、ひと区切り」という感覚で、人が自然に話すときの文の切れ目とだいたい合います。

> 前に触れた「区切り＝メモの単位」を思い出すと、この値が**そのままメモの分かれ方**になります。短くしすぎると細切れのメモが増えるので、ここは実機で話しながら調整しました。

`maxSpeechDuration` は、いわば暴走防止の安全弁です。普段は `minSilenceDuration` が区切ってくれるので、これが効くのは「一度も間を空けずに延々と喋り続けた」ような例外のときだけです。

### 1つ注意：設定を変えたらVADは作り直す

これらのパラメータは **VADを作るときにしか効きません**。あとから値だけ差し替える、ということはできませんでした。

なので、設定を変えたいときは、**古いVADを `free()` で片付けて、新しい値で作り直す**必要があります。Silero VADは軽いので、作り直しても一瞬でした。

## 実際に触ってみた

開発用の画面に「録音開始ボタン ＋ 区切りの設定スライダー ＋ 区切られた発話の一覧」を置いて、試してみました。

録音を始めて、

「今日はいい天気ですね」

としゃべって、少し黙る。すると、リストに1件、

```text
seg#1   1.8 秒
```

のように**発話のかたまりが1個追加**されます。続けて別のことを話すと、また1件増える。

スライダーで `minSilenceDuration` を短くすると、ちょっとした間でもすぐ区切られて細切れになり、長くすると、多少黙ってもひと続きにまとまる——という変化が、その場で確かめられました。

「自分の話したかたまりが、ちゃんと1件ずつ増えていく」のを見ると、地味ですが結構うれしいです。

## つまずき・気になったところ

実際にやってみて、引っかかった点や気づいたことをまとめておきます。

### モデルは実ファイルのパスで渡す

前述のとおり、アセットを直接は渡せず、一度端末にコピーしてからそのパスを渡す必要がありました。最初ここに気づかず、「ファイルが見つからない」で少しハマりました。

### 渡す単位（512サンプル）は決め打ち

VADに渡す量は自由ではなく、Silero VADが要求する512サンプル単位に合わせる必要があります。流れてくる音をいったん貯めて、その単位に切り出して渡す、という一手間が要りました。

### 設定変更はVADの作り直し

パラメータは生成時にしか効かないので、スライダーで調整するたびにVADを作り直す作りにしました。軽いので体感は問題なし。

### この段階では「文字」にはならない

当たり前ですが、VADがやるのは**区切りだけ**です。出てくるのは「音のかたまり」であって、文字ではありません。文字にするには、このかたまりを音声認識モデルに渡す、次の段階が必要です。

## まとめ

「録音した一本の音を、発話ごとに区切りたい」——最初は無音で切るだけかと思っていましたが、ちゃんとやるには **VAD** という“声の門番”が必要でした。

そして、それは `sherpa-onnx` の中に最初から入っていて、**約0.6MBの小さなモデル1つ**を足すだけで動きました。

やったことを整理すると、こんな感じです。

- VADは「いま声がある／ない」を判定する仕組み。中身（何を言ったか）は分からない
- `sherpa-onnx` にVADは内蔵されていて、別ライブラリは不要
- VADモデル `silero_vad.onnx` は約0.6MBと軽く、アプリに同梱できる
- ただしモデルは**実ファイルのパス**で渡すので、初回にアセットからコピーする
- 区切りの箱は **①小数に変換 → ②512サンプルずつ渡す → ③区切りを取り出す** の3ステップ
- 区切り方は `minSilenceDuration`（何秒 黙ったら終わりか）が主役で、これが**メモの分かれ方**そのものになる

音声認識（文字にする部分）に比べると地味な工程ですが、「ずっと聞き続けて、しゃべったら勝手にメモがたまる」ようなアプリでは、この区切りの気持ちよさがかなり体験を左右しそうだと感じました。

次は、この区切ったかたまりを実際に文字にする段階に進む予定です。まだ実験中なので、勘違いがあればぜひ教えてください 🙏
