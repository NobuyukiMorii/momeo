突然ですが、今どきアプリの実装で「音声をテキストにする」って、簡単そうじゃないですか？

僕も最初はそう思っていました。

「適当なパッケージでマイクの音を渡せば、文字が返ってくるんでしょ？生成AIで楽勝っしょ」と。

ところが、自分のアプリに組み込もうとしたら、見事に沼にハマりました。クラッシュするし、遅いし、そもそも「どのライブラリを選べばいいのか」すら分からない。

この週末、音声認識をほぼ初見の状態から一通り触ってみて、ようやく「あー、こういう世界なんだ」と地図が見えてきました。同じように「これから音声認識をアプリに入れたい」と思っている人に向けて、**専門用語をなるべく噛み砕いて**、僕がたどった道をそのまま書いてみます。

## 📌 30秒で結論

忙しい人向けに、先に結論だけ。

- 音声認識エンジンには大きく **「クラウド」** と **「スマホの中（オンデバイス）」** の2種類がある。
- さらに **「しゃべりながら文字が出る（ストリーミング）」** と **「言い終えてから出る（バッチ）」** の違いがある。
- **「ライブラリを入れるだけ」では動かない**。エンジン本体とは別に **モデル（数十〜数百MB）** を用意する必要がある。ここが地味に一番の山。
- 日本語をスマホの中で認識するなら、**`sherpa-onnx` ＋ 日本語特化モデル**がかなり良かった。話し終えて **0.1〜0.2秒**で、句読点付きの自然な日本語が出てくる。
- 速い・正確・軽い、を全部満たす「最強の1つ」は無い。**用途で選ぶ**しかない。

では、順番に。


## 📌 そもそも音声認識ってなに？（1分で）

ものすごくざっくり言うと、**「音 → 文字」に変換してくれる仕組み**です。

マイクで拾った音は、コンピュータの中では「波形の数字の列」になっています。これを「この音の並びは “こんにちは” だな」と当ててくれるのが音声認識。英語だと **STT（Speech To Text）** や **ASR（Automatic Speech Recognition）** と呼ばれます。

中身は、たくさんの音声データで学習した **AIモデル**。「こういう音はこういう文字になりやすい」を学んだ、賢さの本体です。

ここまで分かれば十分です。次は、そもそも僕が何を作ろうとして、この沼にハマったのかを少しだけ。

## 📌 何を作ろうとしていたのか

僕が作っているのは、**「音声ファースト」のメモアプリ**です。

普通のメモアプリだと、「アプリを開く → 録音ボタンを探す → 押す → しゃべる」という手順を踏みます。でも僕が欲しかったのは、**アプリを開いた瞬間からマイクが立ち上がっていて、思いついたことをそのまま口に出すと、勝手に話しかけた内容がテキスト化されて、メモカードがたまっていく**——そんな体験でした。

つまりこのアプリの肝は、**「ずっと聞き続けて、しゃべったらすぐ文字にする」** こと。一言でいうと **常時リスニング** です。

そして、これが今回ハマったポイントの伏線でした。「ボタンを押して短く話す」だけなら実はそんなに難しくないのですが、**「長く・ずっと聞き続ける」となると話が変わってくる**のです。

## 📌 最初の挫折：「OS標準の機能でいいのでは？」

オンデバイスだの何だのに行き着く前に、僕はもっと安直に考えていました。

「**スマホって、もともと音声入力できるよね？ あれを借りればよくない？**」

実際、スマホ（iOS / Android）には OS 標準の音声認識機能があって、Flutter からは **`speech_to_text`** というパッケージで簡単に呼び出せます。最初はこれで実装してみました。

ところが、試しに使ってみたら、**しゃべった内容の2割くらいしか文字にならない**という悲しい結果に。

原因はこうでした。

- OS標準の音声認識は、もともと **「ボタンを押して短く話す」** 用途を想定して作られている。
- 1回の認識は **1発話で勝手に終了** してしまうので、ずっと聞き続けるには「終わったら、また聞き始める」を **ループ** で繰り返すしかない。
- ところが、その **“聞き直す”一瞬の切れ目でマイクがオフ** になり、**その間にしゃべった分がまるごと消える**。連続でしゃべるほど、この切れ目で取りこぼしが積み重なる。

しかも、これはパッケージ公式も「**対象は短いフレーズで、連続認識を実現する手段はまだ無い**」と明言していて、**工夫でどうにかなる問題ではなかった**んですね。

ここでようやく、「OS任せじゃダメだ。音声認識そのものを、ちゃんと自分のアプリの中に持つ必要がある」と腹をくくりました。

そこから先が、この記事の本題です。まずは全体像をつかむために、僕が一番助かった「2つの分かれ道」から見ていきます。

## 📌 迷わないための「2つの分かれ道」

音声認識を調べ始めると、いろんな名前が出てきて混乱します。でも、**まず2つの分かれ道を知るだけで一気に整理**できました。

### 分かれ道①：ネットに送る？ スマホの中でやる？

| | クラウド（ネットに送る） | オンデバイス（スマホの中） |
|---|---|---|
| 精度 | とても高い | 中〜高（端末の性能しだい） |
| ネット | 必要 | **不要（オフライン）** |
| お金 | 使った分だけ課金が多い | **無料**（自分の端末で計算） |
| プライバシー | 音声を外部に送る | **音声が外に出ない** |
| 手軽さ | APIを叩くだけで楽 | モデルの準備などひと手間 |

代表例で言うと、クラウドは Google / Amazon / OpenAI（Whisper API）/ Deepgram など。オンデバイスは後で出てくる sherpa や Vosk などです。

僕のアプリは「プライバシー重視・ずっと聞いていたい・お金をかけたくない」だったので、**オンデバイス**を選びました。

### 分かれ道②：しゃべりながら出る？ 言い終えてから出る？

| | ストリーミング | バッチ |
|---|---|---|
| 文字が出る | **しゃべっている最中から**どんどん | 言い終えた**あとに**まとめて |
| 体感 | リアルタイム字幕みたい | 少し待つ |
| 精度 | やや不利なことも | じっくり処理できて有利 |

スマホの音声入力で、話しながら文字が出てくるあれが「ストリーミング」。一方、録音を止めてから「ポン」と結果が出るのが「バッチ」です。

この2つの軸を持っておくと、どのエンジンの話を聞いても「あ、これはオンデバイスのバッチね」と位置づけられるようになります。

## 📌 ちょっと整理：エンジンの“本体”と、Flutter からの使い方

ここで一回、言葉を整理させてください。僕がいちばん混乱したところです。

音声認識を調べると whisper / sherpa / Vosk みたいな名前がたくさん出てきて、最初は「全部おなじ仲間」だと思っていました。でも整理すると、**“本体”の話と“使い方”の話が混ざっていた**んです。

まず、音声認識エンジンの“本体”は **2つの部品**でできています。

| 部品 | 役割 |
|---|---|
| **モデル（脳）** | 学習ずみの知識の本体（ファイル） |
| **ランタイム（装置）** | モデルを動かして計算するプログラム |

この2つは、速さとメモリ効率が命なので、たいてい **C / C++ で書かれています**（例：`whisper.cpp`、`onnxruntime`）。

ところが、**Flutter（Dart）からは、この C/C++ の本体を直接は動かせません**。そこで、**それを Flutter から呼び出せるようにする「パッケージ」を `pub add` でかぶせます**。これが3つ目の登場人物です。

```
アプリ（Flutter / Dart）
  └ パッケージ              … Flutter から本体を呼び出す“入口”（pub add で入れる）
       └ ランタイム（装置）  … モデルを動かして計算する（C/C++）
            └ モデル（脳）   … 知識の本体（ファイル）
```

ポイントは2つ。

- **「モデル ＋ ランタイム」が音声認識の本体**です（この2つで“エンジン”）。同じランタイムでも、入れるモデルを変えれば日本語特化にも多言語にもなります。
- **パッケージは“本体”ではなく、それを Flutter で使うための入口**。なので3つは横並びではなく、**本体（モデル＋ランタイム）を、パッケージが外側から包んでいる**イメージです。

> ※この記事で「エンジン」と言うときは、ゆるく「モデル＋ランタイムのまとまり」を指します。

この整理ができると、次の登場人物がスッと頭に入ります。

## 📌 各層には、具体的にどんなものがあるの？（誰が作ってる？）

さっきの3層に、実際の名前を入れてみます。「どこが作っているか」が分かると、一気に解像度が上がりました。

### モデル層（脳）の例

| モデル | 開発元 | ひとこと |
|---|---|---|
| **Whisper** | OpenAI（米） | 多言語の定番 |
| **Parakeet / Canary** | NVIDIA（米） | 近年トップ級。速い・正確 |
| **Zipformer 系** | k2-fsa（OSSコミュニティ） | 軽くて速い |
| **SenseVoice** | Alibaba（中） | 中英日韓を1モデルで |

#### ちなみに：モデルは「設計 × データ × サイズ」でできてる

「モデル」と一口に言っても、名前を聞いただけでは中身は決まりません。実は、次の3つの“要素”の組み合わせで決まります。

- **設計**：脳の配線図（例：Whisper / Zipformer / parakeet …）
- **学習データ**：何の音声で勉強したか（例：日本語なら **ReazonSpeech** という大量の音声データ）
- **サイズ**：パラメータ（“脳細胞”）の数。多いほど賢いが、重くなる（例：6億＝0.6B）

だから、同じ設計でも大きさ違い（例：Whisper の small と large）があったり、データや設計を変えれば別のモデルになったり——**同じ「モデル」という言葉でも、中身はこの3要素しだいで変わる**んですね。

### ランタイム層（装置）の例

| ランタイム | 開発元 | 動かせるモデル（対応の広さ） | ひとこと |
|---|---|---|---|
| **whisper.cpp** | ggerganov（OSS） | **Whisper系だけ**（専用） | スマホで速く動かすことに特化 |
| **onnxruntime** | Microsoft | ONNX形式なら何でも（ただAI計算するだけ） | 汎用基盤。下の sherpa が内部で使う |
| **sherpa-onnx** | k2-fsa（OSS） | ONNX形式の**いろんな設計**（Zipformer / parakeet / Whisper / SenseVoice …） | 対応が広い“箱”。モデル差し替えで言語も変えられる |
| **Vosk**（土台は老舗の Kaldi） | Alphacephei | Vosk 用モデル（Kaldi系） | 軽量で、しゃべりながら認識できるオフラインエンジン |

#### ランタイムは「何でも動かせる」わけじゃない（相性がある）

ここで僕がつまずいた疑問。「ランタイムって、どのモデルでも動かせるの？ それとも専用なの？」

答えは **どちらでもなく、「決まった相性がある」**。相性が合うかは、ざっくり**2つ**で決まります。

1. **ファイルの形式が合うか**：モデルには `.jpg` / `.png` のような“形式”がある（ggml形式・ONNX形式・Kaldi形式…）。ランタイムは、自分が読める形式しか開けない。
2. **中の作りに対応しているか**：形式が合っても、ランタイムがその**“作り”を知らないモデルは動かせない**。

ゲーム機で例えると、**①形式＝ディスクの規格**（PS5のディスクはSwitchに刺さらない）、**②作りへの対応＝本体がそのソフトに対応しているか**（刺さっても未対応ソフトは起動しない）。両方そろって、はじめて動きます。

### パッケージ層（Flutter用の部品）の例

最後に、これらの装置（ランタイム）を **Flutter から呼び出すための部品**。基本は「使いたいランタイムに対応した部品を1つ `pub add` する」だけです。

| パッケージ | 呼び出すランタイム（装置） | どんな部品？ |
|---|---|---|
| **whisper_flutter_new** | whisper.cpp | Flutter から Whisper を動かすための部品（有志が開発） |
| **sherpa_onnx** | sherpa-onnx | sherpa-onnx を作っている **k2-fsa 自身が出している公式の部品**。モデルを差し替えれば、日本語特化にも多言語にもできる |
| **vosk_flutter_service** | Vosk（Kaldi） | **Vosk**（＝軽量で、しゃべりながら認識できるオフライン音声認識エンジン。土台は老舗の Kaldi）を Flutter から使う部品。モデルを自動ダウンロードしてくれて手軽 |
| **speech_to_text** | OS標準の音声認識 | スマホ内蔵の音声認識を借りる部品。モデルの準備が要らず手軽だが、**短い発話向け**（記事の最初で挫折したのがこれ） |

## 📌 実機で比較してみた結果

ここまでに出てきたものを、手元のスマホに入れて、同じように話して比べてみました。さっきの「装置（ランタイム）＋ 脳（モデル）」の組み合わせごとに並べています。

| パッケージ | ランタイム（装置） | モデル（脳） | 文字が出る速さ | ストリーミング | 日本語の精度 | モデルの大きさ |
|---|---|---|---|---|---|---|
| whisper_flutter_new | whisper.cpp | Whisper small | 話し終えて **約4秒** | ✗（バッチ） | 高い | 181MB |
| sherpa_onnx | sherpa-onnx | Zipformer | **約0.05〜0.14秒** | ✗（バッチ） | そこそこ | 160MB |
| sherpa_onnx | sherpa-onnx | **parakeet** | **約0.1〜0.28秒** | ✗（バッチ） | **一番良い** | 625MB |
| vosk_flutter_service | Vosk（Kaldi） | Vosk 小モデル | しゃべりながら | ✓（ストリーミング） | 低め | 48MB |

> ※「バッチ」＝言い終えてからまとめて変換。「ストリーミング」＝しゃべっている最中から出る。
> ※ Zipformer と parakeet は、どちらも日本語の音声データ **ReazonSpeech** で学習（parakeet は NVIDIA の NeMo系）。

僕の用途（メモとして読み返す）だと、**速くて・正確で・句読点も付く sherpa-onnx ＋ parakeet が一番しっくり**きました。唯一の弱点は 625MB という大きさで、これを「どう配るか」が次の課題です。

## 📌 で、結局「アプリに組み込む」って何をするの？

「どうやって組み込むの？」に一言で答えると、**やることは大きく4ステップ**です。ひとつずつ見ていきます。

### ステップ1：部品（パッケージ）を入れる

まずは、アプリに音声認識を足すための部品（パッケージ）を `pub add` で入れます。これで、アプリのコードから「音声を認識して」と呼び出せるようになります。

ここでの“あるある”は、**部品同士のバージョンが合わずに弾かれる**、**ネイティブの部品が二重に入ってビルドが落ちる**、といった出来事。技術そのものというより“環境の相性”の問題で、最初の小さな関門になりがちです。

### ステップ2：モデル（脳）を端末に用意する（← ここが一番の山）

実は、部品を入れただけでは話せません。肝心の“脳”＝**モデル**が別に必要だからです。しかもモデルは数十〜数百MBと大きく、**部品には含まれていない**ことが多い。

なので、アプリの初回起動などで**モデルをダウンロードして端末に置く**ことになります。今回の日本語モデルは **625MB** もあって、「これをどうやってユーザーの端末に届けるか（初回ダウンロード／Wi-Fi 限定／空き容量の確認…）」が、正直いちばんの悩みどころでした。

### ステップ3：マイクの音を渡す

あとは、マイクの音（生の波形データ）をモデルに流し込みます。ずっと聞き続ける用途なら、間に **VAD**（＝しゃべってる／無音を判定する部品）をはさんで、**「ひと区切りの発話」にまとめてから**渡すと扱いやすくなります。

ちなみに「ずっと聞いてたら録音が溜まるのでは？」は心配いりません。**音は処理されたら捨てられ、残るのはテキストだけ**。ストレージにもプライバシーにも優しい作りです。

### ステップ4：返ってきた文字を保存する

最後に、モデルから返ってきたテキストを受け取って、メモとして保存（データベースなど）すれば完成です。これで「しゃべったら、文字が残る」が動きます。

---

動いているときは、全体としてこんな一本道になります。

```
🎤 マイク →〔VADで区切る〕→ モデルで文字に変換 → テキストを保存
```

## 📌 そもそも「VAD」って何？

組み込みのステップ3で出てきた **VAD**。地味ですが、“ずっと聞く系”のアプリでは結構な働き者なので、独立して説明します。

### 何をするもの？

**VAD（Voice Activity Detection）＝「いま声がある？ ない？」を判定するだけの仕組み**です。ポイントは、**言葉の中身（何を言ったか）までは分からない**こと。あくまで「声がある／無音」を見分ける“門番”です。

### なんで必要なの？

マイクはオンにしている間、**声だけでなく、無音も生活音もぜんぶ拾い続けています**。これを全部そのまま音声認識エンジンに流すと、

- 無駄な計算が増える（無音まで一生懸命に変換しようとする）
- 雑音まで渡すので、精度も落ちる

そこで VAD が **「ここからここまでが“ひと続きの発話”」と区切って**くれます。すると、

- その“発話のかたまり”だけをエンジンに渡せる
- **「しゃべり終わった」＝「ここで文字にしよう」の合図**にもなる
- 無音のときはエンジンを動かさず、電池や計算の節約になる

### どういう仕組みで判定してるの？

ざっくり言うと、**音を 30 ミリ秒くらいの小さなコマに切って、1コマずつ「これは声っぽい？」を判定**しています。出力は **0〜1 の「声である確率」**。

判定のやり方には、大きく2世代あります。

| | 昔ながら（エネルギー方式） | 最近（AIモデル方式） |
|---|---|---|
| 判定の仕方 | 音の**大きさ**で判断 | 小さな**AI**が「声らしさ」を判断 |
| 雑音への強さ | 弱い（エアコンの音を声と間違えがち） | 強い |
| 代表例 | WebRTC VAD | **Silero VAD** |

ただし「確率がちょっと下がった瞬間に即“終わり”」と判定すると、**息継ぎのたびにブツ切れ**になってしまいます。なので実際は「**何コマか連続で無音が続いたら、はじめて“発話終了”**」と少し待つ工夫が入っています（このさじ加減で、区切りの気持ちよさが変わります）。

### VADにはどんな種類がある？

VAD も1つではなく、いろいろあります。さっきの「エネルギー方式 / ニューラル方式」で分けると、代表的なのはこのあたり。

| VAD | 開発元 | 方式 | ひとこと |
|---|---|---|---|
| **WebRTC VAD** | Google（WebRTC 由来） | エネルギー | 定番の古株。音量で判定するので軽いが、雑音には弱め |
| **Silero VAD** | Silero Team | ニューラル | 軽い・多言語・雑音に強い定番。**今回はこれを採用** |
| **TEN VAD** | TEN Framework | ニューラル | 比較的新しい軽量タイプ |
| **Pyannote** | Hervé Bredin ら（研究由来） | ニューラル | 話者分離なども含む高機能版。重めでサーバ向き |

ちなみにこの4つは横並びではなく、**広く使われている定番は WebRTC VAD（軽さ重視）と Silero VAD（精度重視）の2つ**。Pyannote は「話者分離」など高機能寄りで用途が少し違い、TEN VAD は比較的新しい選択肢、という位置づけです。（ほかに NVIDIA の MarbleNet や商用の Picovoice Cobra なども）

#### 今回採用した Silero VAD

この中で僕が選んだのは **Silero VAD**。オンデバイス向けにちょうど良かったからです。

- **約1.8MB**ととても軽く、30ms のコマを **約1ミリ秒**で判定できる
- **6000言語以上**の音声で学習されていて、**言語を選ばず・雑音にも強い**
- ONNX 形式でオフライン動作（ネット不要）

要するに VAD は **「声の門番」**。中身は読めないけれど、「ここからしゃべってる」「いま終わった」を教えてくれるおかげで、音声認識エンジンに“おいしいところだけ”を渡せるわけです。
