# 常時リスニング（連続認識）の限界に関する調査

## この文書について

- **目的**: `speech_to_text` を使って「常時リスニングして話した内容を自動でメモ化する」アプローチを実装した結果、何が問題になったかを記録する。
- **調査日**: 2026-06-07
- **対象**: `speech_to_text` 7.4.0（iOS = `SFSpeechRecognizer` / Android = `SpeechRecognizer` をラップ）
- **きっかけ**: リスニングフロー（`docs/specs/listening_flow.md`）を実装し、Android 実機（Pixel 8a）で検証したところ、話した内容の **2割程度しかテキスト化されない**という問題に直面した。

---

## TL;DR（結論サマリ）

`speech_to_text` は **「短い区切り発話」向け**に設計されており、**「長く話し続ける常時リスニング」には構造的に向いていない**。

- 1回の `listen()` は1発話で終了し、続けるには **`listen()` を呼び直すループ**が必要になる。
- この **再起動の切れ目でマイクが聞いていない空白**が生まれ、連続して話すと**その間の発話が丸ごと欠落**する。
- パッケージ自身も「連続認識は非対応」と明言しており、これは実装の工夫では**完全には解消できない**。

→ 常時リスニングを本気でやるなら、**別アプローチ（録音して一括でテキスト化する方式）**の検討が必要。

---

## 1. やろうとしたこと

仕様（`listening_flow.md`）に沿って、以下のループを実装した。

```text
listen() で認識開始
↓
話す → onResult でテキストが届く
↓
無音が続く → onStatus='done'（確定トリガー）
↓
確定テキストを保存
↓
listen() を呼び直して再開（ループ）
```

`listen()` は1回で終わるため、**「終わったら呼び直す」ループ**で常時リスニングを擬似的に実現しようとした。

---

## 2. 実機で起きた問題

### 問題A: 再起動の切れ目で取りこぼす（最大の問題）

連続して話すと、テキスト化されるのは**話した内容の2割程度**だった。

原因は、セッションの切れ目にある**「マイクが聞いていない空白時間」**。

```text
onStatus: done           ← セッション終了
（再起動の遅延）          ← この間マイクOFF
listen() 再開            ← 立ち上がりにも時間
onStatus: listening      ← ようやく再開
```

この空白の間に話した分が**丸ごと欠落**する。連続発話では切れ目が何度も発生するため、取りこぼしが積み重なる。

### 問題B: ERROR_RECOGNIZER_BUSY（error 8）の多発

`done` の直後にすぐ `listen()` を呼ぶと、Android の `SpeechRecognizer` が前のセッションを解放しきる前に呼ばれ、**`error_busy`（ビジー）**になる。

- ビジーのまま即再起動を繰り返すと**無限ループ**になり、端末の音声認識サービス自体を詰まらせた（端末再起動が必要になるほど）。
- 対策として「再起動前に遅延を入れる」「busy が続いたら遅延を伸ばす（バックオフ）」を実装したが、**遅延を入れるほど問題A（取りこぼし）が悪化する**というトレードオフに陥った。

### 問題C: `isFinal=true`（最終結果）が安定して来ない

仕様では「確定テキストは最終結果（`isFinal=true`）を採用」と決めていたが、Android では**最終結果が来ずに `done` になるケースが多発**した。

- 対策として「最終結果が無ければ最後の非空の部分結果を採用」に変更し、取りこぼしは多少改善した。
- ただし、これは問題Aの根本解決にはならなかった。

## 3. パッケージ側の見解

公式・コミュニティとも、**連続認識は想定外の使い方**であると明言している。

- 公式（pub.dev）: 「対象ユースケースは**コマンドや短いフレーズ**であって、連続的な書き起こしや常時リスニングではない」
- 公式: 「Android/iOS の音声認識機能では、連続認識を達成する手段はまだ無い」
- Issue #253（連続リスニングの再起動方法）: 同じ `error_busy` 問題が報告されているが、**根本的な解決策は提示されていない**

参考リンク:
- [speech_to_text | pub.dev](https://pub.dev/packages/speech_to_text)
- [Issue #253 — Correct way to restart listener for continuous listening](https://github.com/csdcorp/speech_to_text/issues/253)

## 4. 結論と今後の方向性

### 結論

`speech_to_text`（= OS 標準のストリーミング音声認識）は、**「ボタンを押して短く話す → 確定」**のような区切り利用には向くが、**momeo が目指す「常時リスニングで話し続けた内容をメモ化」には構造的に向かない**。

問題A（再起動の空白による取りこぼし）は、設定調整である程度は緩和できても、**仕組み上ゼロにはできない**。

### 今後の方向性（候補）

| 方式 | 概要 | 備考 |
|---|---|---|
| **録音 → 一括テキスト化** | マイク音声を録音し、区切りのよいところで Whisper 等の STT に一括で送る | 取りこぼしが起きにくい。`speech_recognition_accuracy.md` で触れたバッチ型と同じ発想 |
| **オンデバイス STT エンジン** | 連続認識に対応した別エンジンを使う | 要追加調査 |
| **現方式の継続＋割り切り** | `speech_to_text` のまま、短い発話用途に絞る | 常時リスニングのコンセプトは見直しになる |

→ 次は **「録音して一括でテキスト化する方式」**を軸に、別途調査・検証を行う。
